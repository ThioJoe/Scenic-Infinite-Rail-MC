# CONTEXT.md ‚Äî How the Scenic Infinite Rail Mode pack works

A complete technical reference for the project: the architecture, the shared state, every file, and the algorithms. Written for a developer (or an AI) who needs to understand or modify the pack. For the repository layout and build workflow see `BUILDING.md`. After changing the algorithm or any file this document describes, verify with the integration tests ‚Äî see `tests/README.md` for how to run them, what they cover, and how to add new ones for whatever you just changed.

Sections 1‚Äì10 document the **Java Edition** data pack (the original and richest implementation); **section 11** documents the **Bedrock Edition** port and how the two editions share one codebase.

---

## 1. What it is

A **100% vanilla Minecraft: Java Edition data pack** (no mods, no resource pack, no external dependencies) that turns the game into an endless, relaxing "Slow TV" minecart ride. Inspired by "Slow TV" train journey videos, the player glides over plains, bridges ravines and oceans, and tunnels through mountains, forever. (A Bedrock behavior-pack port built from the same sources is covered in ¬ß11.)

**Branding vs. internals:** the public-facing name is **Scenic Infinite Rail Mode** (shortened where space demands: "Scenic Rail Mode", or "Scenic Rail" ‚Äî the chat message prefix is `[Scenic Rail]`, debug lines use `[SR Debug]`). Everything internal keeps the original `infinite_rail` identity: the function namespace, folder `data/infinite_rail/`, the `ir`/`ir_*` scoreboard objectives, entity tags and score-holder names are all unchanged ‚Äî only user-visible strings (pack names/descriptions, chat prefixes, menu titles, shipped folder/file names) carry the new brand.

The ride starts by itself in a fresh world (or via one command): the player is placed on a self-building, permanently-powered rail line heading **due east forever**, while an algorithm lays smooth track over the procedurally generated terrain ‚Äî bridging valleys and oceans, tunneling through mountains, and hovering a few blocks above the ground the rest of the time. The player sits in a real minecart ‚Äî but not the one on the rails: their cart is glued to an invisible, interpolated **camera seat** that flies a pre-smoothed S-curve computed from the track's own recorded profile, while a hidden **pace cart** rides the physical rails behind them and sets the speed. Slope corners and rail physics never reach the player's eyes, and they mount exactly once per ride (¬ß7g).

Everything is driven by `.mcfunction` files and a single scoreboard. There is no Java, no external process. Target versions: **Java 1.21 through 26.2** (see `pack.mcmeta`).

Key design facts to keep in mind while reading:

- **The world is one-dimensional in travel.** The cart only ever moves in **+X (east)**. Z is fixed (the track never turns). Y is what the algorithm decides.
- **The "column"** is the unit of work: one X-slice of track (a rail, its support below, a light above, and carved air around). The pack builds columns one at a time, ahead of the cart.
- **All shared state lives in the scoreboard, on `.`-named fake players.** Values are held on fake players whose names start with `.` (a convention for "not a real player / internal variable" ‚Äî chosen over Java's traditional `#` because `.` parses on Bedrock's command engine too, keeping the shared files byte-identical across editions). There are no data structures beyond that and a little command storage. **Runtime state lives in the classic `ir` objective; the 30+ tunable settings live in three sidebar-sized objectives** ‚Äî `cfg_terrain`, `cfg_camera`, `cfg_ride` ‚Äî so the Debug menu can put any whole group on the scoreboard sidebar (a vanilla sidebar displays exactly one objective, max 15 rows). A fourth objective, `dbg`, is a display-only mirror for the Debug menu's "Live state" sidebar view (¬ß6.10). (Two Java-only extras: `ir_menu`, a `trigger`-criteria objective that is the books' permission-free click channel, and `ir_click`, the Speed items' used-stat channel ‚Äî ¬ß6.9/¬ß6.10.)

## 2. Data pack anatomy & how Minecraft bootstraps it

The **shipped** Java pack (what `tools/build.mjs` assembles into `dist/java/Scenic_Infinite_Rail_Mode/`) looks like this:

```
Scenic_Infinite_Rail_Mode/
  pack.mcmeta                                   # pack metadata + version compat + overlays
  data/
    minecraft/tags/function/
      load.json                                 # vanilla hook: run on load/reload
      tick.json                                 # vanilla hook: run every tick
    minecraft/function/
      ir_*.mcfunction                           # one-line call bridges for the shared
                                                # brain files (bare-name -> namespaced)
    infinite_rail/function/
      *.mcfunction                              # all the logic (namespace: infinite_rail)
    infinite_rail/predicate/
      night.json, day.json                      # vanilla time_check day/night pair
                                                # (torch auto's clock read -- ¬ß6.7)
  overlay_snake/                                # version overlay: replaces files on format 92+
    data/infinite_rail/function/
      setup_world.mcfunction                    # snake_case gamerules (26.x)
      names.mcfunction                          # snake_case command/gamerule names (26.x)
    data/infinite_rail/predicate/
      night.json, day.json                      # time_check with the 26.1+ required
                                                # clock field (minecraft:overworld)
```

**In the repository**, these files are split across `src/java/` and `src/shared/functions/`: ten functions (`config`, `modes_init`, `consts`, `decide`, `consider_start`, `start_event`, `end_event`, `speed_step`, `torch_auto`, `debug_state`) are *shared source* used **byte-identically** by both the Java pack and the Bedrock port, and the build drops them into `data/infinite_rail/function/` alongside the Java-only files (see `BUILDING.md` and ¬ß11). The `data/minecraft/function/ir_*` files are the Java half of the shared files' call bridges (¬ß11a). Nothing about the shipped pack differs from the layout above.

**Version overlay.** `pack.mcmeta` declares an *overlay* (`overlay_snake`) that applies on data-pack **format 92+** (25w44a onward, the 26.x "snake_case gamerule" era). Files inside it transparently **replace** the same-path files in `data/` on those versions, so the base pack carries the camelCase (format 82-91) copies and the overlay carries the snake_case ones. The shared logic just calls `setup_world` / `names` once and always gets the version-correct copy ‚Äî no runtime branching, no compile-drop, no duplicate calls. (Overlay format numbers: 92 = 25w44a's rename; 107 = 26.2 ‚Äî bump the overlay `max_format` alongside the pack's when extending support.)

Minecraft discovers a data pack by its `pack.mcmeta`. Two **vanilla function tags** are the only entry points the game calls on its own:

- `#minecraft:load` ‚Üí lists `infinite_rail:load`. The game runs it **once when the world loads and again on every `/reload`.** This is where the pack initializes.
- `#minecraft:tick` ‚Üí lists `infinite_rail:tick`. The game runs it **every game tick (20√ó/second).** This is the pack's heartbeat.

Everything else is a normal function reached by `function infinite_rail:<name>` calls, or by the player running `/function infinite_rail:start` / `:stop`.

> **Important behavior:** the game loads every `.mcfunction` into memory at load/`/reload` time. Editing a file on disk does **not** change the running game until `/reload` (or a world rejoin). This is why `config` is applied via `/reload`, not by re-running the `config` function (see ¬ß6).

## 3. Coordinate & geometry conventions

- **+X = east = the direction of travel.** The head advances in +X.
- **Z is constant** ‚Äî the centerline of the track. It never changes after start.
- **Y** is the elevation the algorithm chooses per column.
- **The head marker** (`ir_head`, ¬ß4) sits at the current build position: `(headX + 0.5, railY, centerZ + 0.5)` ‚Äî block-centered in X/Z, integer Y. Most build commands `execute ... at @e[ir_head]` and then use `~` relative coordinates, so in the place/support/sample functions:
  - `~` = the rail's cell (Y = railY)
  - `~-1` = one below the rail (the support / redstone block)
  - `~3` = three above the rail (the light block)
  - `~4` / `~5` = top of the carved clearance
  - `~-8 .. ~8` in Z (forceload) = ¬±1 chunk around the centerline (widened
    up to `.TORCHRANGE` while torch mode is on or auto ‚Äî see `forceload_here`)

A single **column** therefore looks like this vertically (flat case):

```
  railY+4 .. railY+1   air (carved clearance / tunnel bore)
  railY+3              minecraft:light   (lights tunnels, blocks mob spawns; the level is the
                       Track light mode .LIGHTMODE -- 11 bright/default, 8 dim, 0 = no block at all)
  railY                minecraft:powered_rail (always powered)
  railY-1              minecraft:redstone_block   (powers the rail; disguised as smooth_stone by a block_display)
```

Consecutive columns differ in X by 1. On slopes they also differ in Y by 1, producing a 45¬∞ "corner-to-corner" line of ascending rails (see ¬ß7c).

The carve is **vegetation-sparing** (¬ß7i): only the rail cell and the cell above it (center) are cleared unconditionally; the side cells and the center cells ‚â• 2 above the rail leave natural vegetation standing (terrain always carves). Slope columns and the `.SLOPECLEAR` columns around them clear their full center bore regardless.

## 4. Shared state

### 4.1 The scoreboard objectives

All variables sit on `.`-named fake players (both editions, same spelling), spread over a small family of `dummy` objectives:

- **`ir`** ‚Äî every runtime/state score (plus `.DEBUGMODE` and `.AUTOSTART`, the two knobs not worth a sidebar row).
- **`cfg_terrain`** (15 rows ‚Äî at the sidebar cap, full) ‚Äî terrain sampling / slope shaping: `.HOVER` `.TUNNELCLEAR` `.MIN_CHANGE` `.SAMEGAP` `.TURNGAP` `.GAPRATIO` `.GAPMATCH` `.SLOPECLEAR` `.DOWNCLAMP` `.UPGRACE` `.UPEARLY` `.DOWNLOOK_AHEAD` `.DOWNGRACE` `.SAMPLE_WINDOW` `.SAMPLE_BLOCK_INTERVAL`
- **`cfg_camera`** (7 rows) ‚Äî the ride rig: `.CAMHEIGHT` `.CAMBLEND` `.CAMSMOOTH` `.CAMLIFT` `.RIDER_BEHIND` `.CAMMODE` `.CARTYOFF`
- **`cfg_ride`** (14 rows) ‚Äî speed, mode knobs, performance (plus one terrain-shaping guest, `.GAPSTRETCH` ‚Äî cfg_terrain is at the 15-row sidebar cap): `.DEFAULTSPEED` `.OCEANSPEED` `.OCEANCHUNKS` `.LANDCHUNKS` `.SKYY` `.SKYSPEED` `.TORCHODDS` `.TORCHRANGE` `.SEAPICKLE` `.GAPSTRETCH` `.CARTSOUND` `.PACE_CART_BEHIND` `.TERRAIN_GENAHEAD` `.BUILD_PER_TICK`
- **`dbg`** ‚Äî display-only: the Debug menu's "Live state" sidebar mirror (¬ß6.10). Never read by logic.

The three `cfg_*` groups exist because a vanilla scoreboard sidebar displays **one objective at a time, max 15 rows** ‚Äî grouped this way, the Debug menu can show any complete settings group (or the live state) on the sidebar. Both editions create all objectives at load (Java `load.mcfunction`, Bedrock `init()`), always before `config` runs. In `/scoreboard` commands, use the objective named in the tables below (e.g. `/scoreboard players set .HOVER cfg_terrain 8`).

**Tunable config knobs** (set by `config.mcfunction` into their `cfg_*` objective ‚Äî the lists above; see ¬ß8):

| Score        | Meaning |
| ------------ | ------- |
| `.HOVER`     | Preferred rail clearance (blocks) above the average terrain surface. |
| `.TUNNELCLEAR`    | Clearance bore height (blocks above the rail) carved per column; the tunnel/headroom height. Slope columns carve `.TUNNELUP` (= `.TUNNELCLEAR+1`). Keep ‚â• 3 (the light sits at rail+3). |
| `.DEFAULTSPEED`  | Default value pushed into the minecart max-speed gamerule at ride start (blocks/s). Applied once, not enforced. Needs the Minecart Improvements feature to have any effect. |
| `.OCEANSPEED`| The **default** speed while crossing open ocean (frozen oceans count as land): the seed for the adjustable ocean cruise `.ocnspd` (copied once by `modes_init`) and what the Speed Reset item returns it to. While the sprint is on the ride cruises at `.ocnspd`, which the Speed items tune in **both** directions (the old max-with-the-land-speed rule is retired). `0` disables the ocean speed-up entirely. |
| `.OCEANCHUNKS`| Consecutive ocean-biome chunks the ride must cross before speeding up to `.OCEANSPEED`. |
| `.LANDCHUNKS`| Consecutive non-ocean chunks after a speed-up before reverting to `.DEFAULTSPEED`. |
| `.SKYY`      | Sky mode's fixed cruising altitude: while `.SKYMODE` is 1 the shared `decide` steers the rail to exactly this Y (¬ß6.9). Raise it toward ~260 to clear even the tallest jagged peaks. |
| `.SKYSPEED`  | Sky mode's **default** cruising speed (blocks/s): the seed for the adjustable `.skyspd` state score (copied once by `modes_init`) and the value the Speed Reset item returns the sky cruise to. While sky mode owns the speed the ride cruises at `.skyspd` (default this), which the Speed ‚àí/+/Reset items tune live. |
| `.TORCHODDS` | Torch mode: the **default** percent chance (0-100) per new column of planting a torch beside the line. Only the seed for the `.torchdens` state score (copied once by `modes_init`); the Visual Settings menu's density presets own the live value afterwards (¬ß6.9). |
| `.TORCHRANGE`| Torch mode: the farthest a torch may land from the centerline ‚Äî each torch rolls uniform 2..this (clamped 2-48). Above 8, `forceload_here` widens the Java corridor so the whole band stays loaded ‚Äî while torch mode is on **or auto** (deliberately clock-blind: the chunk pipeline never reads the world clock ‚Äî ¬ß6.7). |
| `.SEAPICKLE` | Torch mode: where a torch would land **on water**, plant a sea pickle on the bed instead (a torch can't stand on water). Value 0-4 = the pickle count = brightness (1=light 6 ‚Ä¶ 4=light 15); `0` = plant nothing (the old skip-water behavior). Default 4. Always on as part of torch mode; there is **no** Settings-menu toggle for it. |
| `.CARTSOUND` | The minecart sound's **default** (1 = on): the seed for the `.SOUNDMODE` state score (copied once by `modes_init` via the `.sndinit` one-shot flag); the Ride Settings menus' Sound switch (`mode_sound_on/off`) owns the live value afterwards (¬ß6.9). The ride cart never rolls on rails on either edition, so the engine plays no riding sound of its own ‚Äî both editions re-create it by playing the vanilla first-person riding sample (`entity.minecart.inside`) at the rider on the same 115-tick clock (the sample's length): Java `/playsound`s it at a large volume so it can't fade (¬ß6.9), Bedrock re-triggers the RP's attenuation-free `ir.cart_roll` definition (¬ß11a/¬ß11e). |
| `.DEBUGMODE` | `1` = print chat messages about the speed system (default applied, each ocean/land chunk with counters + the cart's real speed, every speed change); `0` = silent. |
| `.CAMHEIGHT` | **Extra** rig height above the rail line, in **tenths of a block** (0 = the ride cart rests on the smoothed line like a cart on a rail). Keep it small (<= ~5) so climb corners can't lift your head into tunnel roofs. |
| `.CAMBLEND`  | S-curve blend length in blocks (even): the camera transitions level‚áÑparallel over exactly this distance at every slope change. |
| `.CAMSMOOTH` | Descent glide divisor: the camera closes `1/.CAMSMOOTH` of a **downward** gap per tick (climbs use the constructed S-curve instead; 1 = off). |
| `.CAMLIFT`   | Climb float / crest budget, in **tenths of a block**: how high the camera rides above the rail line while climbing, and how early it reaches the summit level. |
| `.RIDER_BEHIND` | How many blocks **behind the build head** the rig (viewer) rides ‚Äî every distance knob measures from the head. The rig leads the pace cart by `.PACE_CART_BEHIND ‚àí .RIDER_BEHIND` ("the rig lead", 64 at defaults). Keep it below `.PACE_CART_BEHIND` (load warns otherwise) with a lead of at least ~40. |
| `.CAMMODE`   | **Bedrock-only** (inert on Java): `0` = native free-look rig, `1` = eased cinematic camera via Bedrock's camera system (¬ß11). |
| `.CARTYOFF`  | **Bedrock-only** (inert on Java): fine-tune for the minecart visual's height, in tenths of a block (negative = lower). The base correction is baked into the pack's re-based model copy (`geometry.ir_cart`, 16px down -- vanilla's cart geometry draws a block high outside the engine's internal renderer), so keep this small. Live-tunable mid-ride. Ignored while `.HIDECART` is on (hide-minecart mode glides the prop at its own fixed ‚àí0.5-block sink instead ‚Äî ¬ß6.9). |
| `.AUTOSTART` | `1` = the ride auto-starts for the first player in a fresh world; `0` = manual start only. |
| `.MIN_CHANGE`  | Minimum `|target ‚àí railY|` before a slope change is even considered (hysteresis vs. terrain noise). |
| `.SAMEGAP`   | Minimum flat columns between two elevation changes **in the same direction**. Shortened after a large event by the big-event gap credit (¬ß7k). |
| `.TURNGAP`   | Minimum flat columns before the rail may **reverse** direction. Shortened after a large event by the big-event gap credit (¬ß7k). |
| `.GAPRATIO`  | The big-event gap credit's ratio (¬ß7k), as a **percent** of the last event's height (√ó100 fixed point, so fractional ratios work on the int-only scoreboard): the next event's required gap is reduced by `height √ó this / 100`. `50` = a 40-block climb shortens the next gap by 20 columns; `67` ‚âà two thirds (a 1/1.5 ratio); values above 100 over-credit; `0` = credit off, the gaps always apply in full. |
| `.GAPMATCH`  | The credit's worth-it guard (¬ß7k), also a **percent** of the last event's height: the newly wanted move must itself be at least `height √ó this / 100` blocks, or the full gap applies. `50` = a follow-up at least half the big event's size qualifies; `0` = no size requirement. |
| `.GAPSTRETCH`| The stretch shift's required landing-stretch length, in columns (¬ß7l): a gap-blocked **descent** may jump its gap entirely when the shift scan verifies the whole planned 45¬∞ path clear of ground **and** this many columns of ground sitting at the landing level. `0` = off. Lives in `cfg_ride` (cfg_terrain is full). Keep the descent depth + this ‚â§ ~96 (the scan's reach) for the shift to be verifiable. |
| `.SLOPECLEAR`| How many columns just **before and after** every slope get their full-height center clear even through vegetation (¬ß7i) ‚Äî the camera floats above the rail line around slopes. Vertical only; the cells left/right of the track always spare plants. Keep ‚â• the camera's lift-off run (~`.CAMBLEND/2 + .CAMLIFT/10 + 2`) and ‚â§ `.SAMEGAP`. 0 = only the slope columns themselves. |
| `.DOWNCLAMP` | Max a single heightmap sample may pull the rolling average **down** per column. Smaller values mean ravines and canyons are ignored and bridged dead level instead of dipped into. |
| `.UPGRACE`   | How many blocks **above** the average-derived `.target` a climb may overshoot to clear ground the climb-side scan (always the full `.SAMPLE_WINDOW` ‚Äî ¬ß7j) still sees near the rail line (crest completion ‚Äî wide hilltops are ridden over at hover height instead of tunneled just under the summit; narrow ridges above this budget still get punched). `0` = climbs stop exactly at the target. |
| `.UPEARLY`   | The climb schedule's slack (blocks): how much sooner than *strictly necessary* a climb may begin. The scan projects every surface ahead onto a 45¬∞ line; a climb is held until the rail is within this many blocks of the projected height (¬ß7j). `0` = ramps start at the last possible column and top out right at the crest; bigger = earlier, longer ramps that finish about this early; ~50+ = no schedule (the old ramp-up-way-early behavior). |
| `.DOWNLOOK_AHEAD`  | Descent-side ground scan reach (blocks ahead of the head, hard-capped at `.SAMPLE_WINDOW` ‚Äî ¬ß7j): a down-step is only taken when the rail stays above the **tallest** surface in this range (+`.DOWNGRACE`), so descents can never trench ‚Äî a blocked descent ends just above the ground and continues, `.SAMEGAP`-paced, once it falls away. Also the clear-runway requirement: dips *narrower* than this are crossed level (bridged) instead of dipped into. Bigger = a calmer line that only descends into wider openings; smaller = hugs every hollow. `0` = descent timing is ruled by the average alone. |
| `.DOWNGRACE` | Clearance a descending step keeps above that tallest scanned surface. `0` = a descent may touch down exactly onto the highest nearby ground; higher = stops descents sooner / flies higher over crossed terrain. Keep it `< .HOVER`, or descents end just short of their target even over flat ground and the line rides permanently high. |
| `.PACE_CART_BEHIND` | How many blocks **behind the build head** the hidden pace cart rides ‚Äî equivalently, how far ahead of the cart the rails are kept built (Java: keep < ~250; Bedrock: useful up to ~270, the single-scout ceiling). Keep it above `.RIDER_BEHIND`. |
| `.TERRAIN_GENAHEAD` | **Java only**; how far (blocks) ahead of the **build head** terrain is force-generated. Keep it comfortably above `.SAMPLE_WINDOW` (load warns otherwise). |
| `.BUILD_PER_TICK` | Max columns built per game tick (catch-up budget). |
| `.SAMPLE_WINDOW` | How far ahead of the build head the terrain-averaging window reaches (blocks) ‚Äî the line's planning horizon. Also the near-ground scan's reach (and `.DOWNLOOK_AHEAD`'s cap). Keep ‚â§ `.TERRAIN_GENAHEAD`. |
| `.SAMPLE_BLOCK_INTERVAL` | The spacing between terrain readings inside the window; the per-column sample count (and the average's divisor, `.winn`) is `.SAMPLE_WINDOW / this` ‚Äî 12 at the defaults 48/4. |

**Mode toggles** (state, not config: flipped by the `mode_*` functions ‚Äî ¬ß6.9 ‚Äî seeded to 0 by the shared `modes_init` with add-0 (the non-zero defaults seed via one-shot flags instead: `.TORCHMODE` starts at 2 = auto via `.trchinit`, `.AGGROMODE` at 1 via `.agginit`, `.LIGHTMODE` at 11 via `.lgtinit`, `.SOUNDMODE` copies the config default `.CARTSOUND` via `.sndinit`), and deliberately NOT reset by `config`/`/reload`; like every `ir` score they persist in the world save):

| Score        | Meaning |
| ------------ | ------- |
| `.RAINMODE`  | 1 = permanent rain is on. Informational once set ‚Äî the weather-cycle gamerule and `/weather` do the actual work. |
| `.NIGHTMODE` | **Tri-state** time mode: `0` = default day/night cycle, `1` = Always Night (frozen midnight), `2` = Always Day (frozen noon). Informational once set ‚Äî the daylight-cycle gamerule and `/time` do the actual work. |
| `.TORCHMODE` | **Tri-state** torch scatter: `0` = off, `1` = always on, `2` = **auto** (the default ‚Äî torches only while the world clock says night; the shared `torch_auto` answers `.torchlit` per column). While lit, each edition's builder plants torches beside new columns (Java `place_torch`/`torch_try`, Bedrock `maybeTorch()`), or a sea pickle on the bed where a torch would land on water (config `.SEAPICKLE`). |
| `.SKYMODE`   | 1 = sky cruise: the shared `decide` overrides `.target` with `.SKYY`, and the editions pin the speed to the **sky cruise `.skyspd`** (adjustable, default `.SKYSPEED`) while pausing the ocean system. |
| `.speed`     | The **adjustable LAND ride speed** (blocks/s, floored at 1 by the shared `speed_step` ‚Äî deliberately no upper cap; Java tops out wherever vanilla bounds the minecart max-speed gamerule, Bedrock's virtual pace takes anything but eases off when the builder can't keep up): what `begin` applies at start and `speed_down`/`mode_sky_off` restore. Seeded from `.DEFAULTSPEED` by `modes_init` the first time; nudged `.SPEEDSTEP` (4) blocks/s per click of the Speed ‚àí / Speed + hotbar items (a +click from the clamp floor of 1 rejoins the grid at 4, not 5), reset to the default by the Speed Reset item / Ride Settings menu. State like the modes ‚Äî survives `/reload`, ride restarts and rejoins. **The Speed items tune whichever cruise is active: `.skyspd` in sky mode, `.ocnspd` during an ocean sprint, `.speed` otherwise ‚Äî the other two are untouched.** |
| `.skyspd`    | The **adjustable SKY cruise speed** (blocks/s) ‚Äî what the ride cruises at while `.SKYMODE` is 1. Seeded from config `.SKYSPEED` by `modes_init` the first time (exactly like `.speed` from `.DEFAULTSPEED`), then tuned by the Speed ‚àí/Reset/+ items **while sky mode is on** through the same shared `speed_step`. So sky mode "jumps to its default" on first use but is adjustable from there, and a chosen sky speed persists across `/reload`, ride restarts and rejoins like every mode. Java's `sky_speed` applies it to the gamerule; Bedrock's `tickPace` reads it as the pace target. |
| `.ocnspd`    | The **adjustable OCEAN cruise speed** (blocks/s) ‚Äî what the ride cruises at while the ocean sprint is on (`.fast` 1). Seeded from config `.OCEANSPEED` by `modes_init`, tuned by the Speed items **while the sprint is active, in both directions** ‚Äî the old `max(.OCEANSPEED, .speed)` "the ocean never slows the ride" rule is retired in its favor. Java `speed_up` re-applies it every ocean chunk (and `speed_apply` pushes a mid-sprint click immediately); Bedrock's `tickPace` asserts it every tick while `S.fast`. Persists like every mode. |
| `.LIGHTMODE` | The **Track light** level: the light block placed 3 above every **new** rail (`place_light`/Bedrock `placeColumn`). Presets: 11 = bright (the default ‚Äî also the ice-melt-safe maximum), 8 = a dim glow, 0 = no light at all (dark tunnels and nights; hostile mobs can spawn in the unlit tunnels left behind). Any hand-set 1-15 works on Java (and on Bedrock in API-bridge mode; cmd-bridge reads degrade to the nearest preset). Set by `mode_light_on/low/off` or the Visual Settings menus; new columns only ‚Äî built track keeps its lights. Non-zero default, seeded via the `.lgtinit` one-shot. |
| `.torchdens` | The **torch-mode density** (percent chance per column) the placement roll actually uses (Java `place_torch`, Bedrock `maybeTorch()`). Seeded from `.TORCHODDS` by `modes_init` the first time; set to 15/35/70/100 by the Visual Settings menu's Low/Medium/High/Max presets (the `torch_density_*` functions ‚Äî users only ever see the friendly names). State like `.speed` ‚Äî survives `/reload`, ride restarts and rejoins. |
| `.HIDECART`  | 1 = **hide the minecart**: Java removes the ride cart and seats the rider directly on the invisible camera seat (`mode_hidecart_*` + the keeper/`launch_done` branches); Bedrock keeps the scenery cart prop but glides it at a fixed ‚àí0.5-block sink (`HIDE_CARTYOFF`, in place of `.CARTYOFF`) below the track line, out of the rider's view. |
| `.SOUNDMODE` | 1 = the **minecart sound** plays while riding: on Java `main`'s 115-tick clock re-triggers `/playsound entity.minecart.inside` at the rider (`sound_loop`), on Bedrock `tickSound()` re-triggers the RP's `ir.cart_roll` copy of the same sample on the same 115-tick cadence (¬ß6.9/¬ß11e). Unlike the pure toggles it has a config default (`.CARTSOUND`), seeded once by `modes_init` via the `.sndinit` one-shot flag; the Ride Settings menus and `mode_sound_on/off` own it afterwards. Independent of `.HIDECART` (a hidden cart can still be heard). |
| `.AGGROMODE` | 1 = **mobs aggro** (the default, seeded via the `.agginit` one-shot): hostile mobs can see the rider and react ‚Äî sneak up, hiss, draw bows ‚Äî pure ambience, the rider is untouchable regardless. 0 = an **invisibility effect on the rider** makes mobs ignore the ride (Java: sight range shrinks to a few percent; Bedrock: invisible players are completely undetectable). The effect is applied natively: Java `mode_aggro_*`/`launch_done`, Bedrock the keeper (once a second). On Bedrock the same effect is what hides the first-person arm ‚Äî the retired `.HIDEHAND` knob's old job ‚Äî so aggro off hides the arm and aggro on shows it (¬ß6.9, ¬ß11e). |
| `.SIDEBAR`   | Which sidebar view the Debug menu selected: `0` = hidden, `1`-`3` = `cfg_terrain`/`cfg_camera`/`cfg_ride`, `4` = the live-state mirror (gates the per-tick `dbg` refresh). |

**Internal constants** (set by `load.mcfunction`, or ‚Äî for the cross-edition ones ‚Äî by the shared `consts.mcfunction`; kept out of user config):

| Score   | Meaning |
| ------- | ------- |
| `.C2`,`.C10` | Small divisors for the camera scan geometry (`.CAMBLEND/2`, `.CAMLIFT` tenths‚Üíblocks). |
| `.C16`  | Blocks per chunk (**16**) ‚Äî the divisor for the ocean-biome chunk counter. |
| `.C100` | Fixed-point multiplier/divisor **100**, from the shared `consts.mcfunction` (both editions): the percent divisor for the big-event gap credit's `.GAPRATIO`/`.GAPMATCH` math in the shared `consider_start` (¬ß7k); Java's camera math reuses it to convert `.CAMHEIGHT`/`.CAMLIFT` (tenths of a block) to milliblocks. |
| `.C1000`| Fixed-point multiplier **1000**: converts whole blocks to milliblocks / extracts the cart's sub-block X fraction. |

| `.SPEEDSTEP` | How much one click of the Speed ‚àí/+ hotbar items changes the ride speed (blocks/s, **4**). Set by the shared `consts.mcfunction` ‚Äî both editions run it at load ‚Äî and deliberately not a config setting (¬ß6.10). |

## 6. File-by-file reference

### 6.1 Metadata & vanilla hooks

**`infinite_rail/pack.mcmeta`**
Pack metadata. Declares the description and version compatibility with the current (25w31a+) scheme: `pack_format` (`84`), `min_format` (`82`) / `max_format` (`107`) ‚Äî the supported *data-pack* format range (25w31a-era through 26.2; a **separate series** from resource-pack numbers). Also:
- `features.enabled: ["minecraft:minecart_improvements"]` ‚Äî **the pack itself turns on the Minecart Improvements feature**, so the minecart max-speed gamerule always exists while the pack is loaded (no manual experiment toggle needed for `.DEFAULTSPEED` / the ocean speed-up).
- `overlays.entries` ‚Äî one overlay, `overlay_snake`, for `min_format` 92 / `max_format` 107. On those versions (25w44a+, snake_case gamerules) the files in `overlay_snake/` replace the base copies (see ¬ß2). The `formats` field is omitted deliberately: it's only required when an overlay range dips below format 82, and this pack's floor is 82.

**`data/minecraft/tags/function/load.json`**
Vanilla tag `#minecraft:load`; its `values` list contains `infinite_rail:load`. Makes the game run `load` on world-load and `/reload`.

**`data/minecraft/tags/function/tick.json`**
Vanilla tag `#minecraft:tick`; lists `infinite_rail:tick`. Makes the game run `tick` every game tick.

### 6.2 Initialization & config

**`function/load.mcfunction`**
Runs on load/reload. Creates every objective (idempotent): `ir` plus the three settings groups `cfg_terrain`/`cfg_camera`/`cfg_ride` (¬ß4.1), `dbg` (the Live state sidebar mirror ‚Äî ¬ß6.10), `ir_menu` (the books' `trigger`-criteria click channel ‚Äî ¬ß6.9) and `ir_click` (the Speed items' `minecraft.used:minecraft.carrot_on_a_stick` stat channel ‚Äî ¬ß6.10); sets the internal constants `.C16 = 16`, `.C100 = 100`, `.C1000 = 1000` (the old `.C12` sample count is gone ‚Äî `sample_window` derives it per column into `.winn`); runs the shared `consts` (the cross-edition internal constants, e.g. `.SPEEDSTEP` ‚Äî ¬ß6.10); calls `infinite_rail:config` to apply all tunables; seeds the mode toggles and the adjustable ride speed via the shared `modes_init` (add-0, so an enabled mode ‚Äî and a chosen `.speed` ‚Äî survives the reload ‚Äî ¬ß6.9); derives `.TUNNELUP = .TUNNELCLEAR + 1`; **warns on the distance-knob ordering invariants** (`.SAMPLE_WINDOW ‚â§ .TERRAIN_GENAHEAD`, `.RIDER_BEHIND < .PACE_CART_BEHIND` ‚Äî violations only degrade the ride, so they get a loud chat line instead of silence); calls `names` to load the version-correct command/gamerule names (the minecart-speed gamerule name into storage `infinite_rail:speed rule`, the weather-/daylight-cycle and chain-budget names into `infinite_rail:names`); **raises the per-chain command budgets** (`maxCommandChainLength` / `maxCommandForkCount` ‚Üí 1,000,000, via `set_rule` with the version-correct names) ‚Äî vanilla caps one command chain at 65,536 commands, and the synchronous ride start pre-builds rig-lead+32 columns (each with a 24-probe near scan, ¬ß7j) in a single chain, which silently truncated `begin` mid-way at the default cap (track built, rider never mounted, `.started` never set); prints a "Loaded" message. Does **not** touch ride state (including `.autodone`), so a `/reload` mid-ride refreshes the knobs without stopping it, and a stopped world stays stopped.

**`function/config.mcfunction`** *(shared source: `src/shared/functions/`)*
The single file a user edits ‚Äî and the byte-identical source file the Bedrock port runs (¬ß11a). Sets every tunable score into its `cfg_*` objective (the three groups in ¬ß4.1; only `.DEBUGMODE` and `.AUTOSTART` go to `ir`) with heavily-commented explanations. The mode *toggles* themselves (`.RAINMODE` & co.) deliberately do NOT live here ‚Äî a reload re-runs this file and would shut every mode off (see `modes_init`, ¬ß6.9). Called by `load` (which then derives `.TUNNELUP`). Its header documents how to apply edits (`/reload`) and that running `config` by itself only re-runs the in-memory copy (so it's only good for resetting live `/scoreboard` tweaks).

### 6.3 Lifecycle / control

**`function/start.mcfunction`**
The player entry point. `execute as @p at @s align xz run function infinite_rail:begin` ‚Äî runs `begin` as the nearest player, positioned at that player's block (X/Z floored to the grid, so the head marker lands block-aligned).

**`function/begin.mcfunction`**
Sets up and launches a ride (see the flow in ¬ß5). Notable steps:
- **Reset:** `.started=0`, `.autodone=1` (a ride has now been started in this world ‚Äî the auto-starter must never fire again), kill any `ir_head`/`ir_probe`/`ir_cart`/`ir_seat`, `forceload remove all`, dismount the player ‚Äî so `start` is safely re-runnable.
- **World tuning:** calls `setup_world`; pre-unlocks every recipe for the starting player (`recipe give @s *` ‚Äî Java has no recipe-unlocking gamerule, so with nothing left to unlock no "recipes unlocked" toast can ever pop mid-ride; costs one combined toast at start. Bedrock instead turns unlocking off in its `setup_world`); applies the default minecart max-speed (`.DEFAULTSPEED` via the `set_speed` macro) and clears the ocean fast state (`.fast = 0`); if sky mode was left on, re-applies `.SKYSPEED` over the default (`sky_speed`, ¬ß6.9).
- **Anchor:** summons the two markers at the player (`~0.5 ‚Ä¶ ~0.5` = block center); force-loads a small area behind + the `.TERRAIN_GENAHEAD` corridor ahead (via `forceload_here` ‚Üí the `forceload` macro).
- **Initial elevation:** snaps `ir_probe` onto the surface here (`probe_surface` ‚Äî the heightmap plus the not-terrain dig-down, so starting on a roof or under a tree still anchors to the ground), reads its Y into `.railY`, adds `.HOVER`, and writes that Y into the head marker via storage `tmp.y`.
- **Init counters:** `.slope=0`, `.flat=99` (large, so the first change isn't gap-blocked), `.lastDir=0`; seeds `.avg = .railY ‚àí .HOVER`; sets `.nextLoad`.
- **Track history:** empties storage `infinite_rail:track y`, sets `.trackBase = .headX` and records the first column's rail Y (index 0).
- **Pace cart:** first **lifts the starting player onto the head marker** (`tp @s` at the head) ‚Äî the anchor *is* their position, and with the default `.HOVER` (2) the first column's solid support block would otherwise land exactly in their head cell: a few ticks of suffocation damage (and the player-hurt sound) before `launch_done` can mount them. Standing in the rail's own cell, the support solidifies under their feet instead. Then places the first column (`place_flat`), summons `ir_cart` (invuln, small eastward motion) and `ir_plug`, and plugs the cart. Seeds the ocean-check state: `.lastChunk` = the rider's starting chunk (pace cart chunk + the rig lead), `.oceanRun` and `.landRun` = 0.
- **Launch handoff:** tags the player `ir_rider`, sets `.pregoal = .headX + (.PACE_CART_BEHIND ‚àí .RIDER_BEHIND) + 32` and `.started = 2` ‚Äî and returns. The runway pre-build and the rig used to run right here, synchronously, but a single command chain that builds ~a hundred columns (each with a 24-probe near scan) silently exceeds vanilla's per-chain command/fork budgets, truncating `begin` before the rig or the mount ‚Äî so the launch is now **phased** through the tick loop (`launch_tick` ‚Üí `launch_done`, below).

**`function/launch_tick.mcfunction`**
The launch driver, run from `tick` while `.started` is 2: samples `.cartX`, sets `.budget = 24` and runs `build_loop` ‚Äî up to 24 runway columns per tick, each tick its own fresh command chain, so the launch can never be truncated by the per-chain budgets no matter how heavy the per-column pipeline or the config gets. Once `.headX ‚â• .pregoal`, runs `launch_done`.

**`function/launch_done.mcfunction`**
Finishes the launch: summons `ir_seat` (`teleport_duration:1`) and `ir_ride` (invuln, yaw 90) **at the rider** and mounts the stack at distance zero ‚Äî ride cart onto seat, then the player **into the ride cart**, the one and only player mount of the ride (while `.HIDECART` is 1 the ride cart is skipped entirely and the player mounts the bare seat instead ‚Äî ¬ß6.9) (mount events flash the client's un-hideable "press ‚áß to dismount" hint, so they must never repeat). Adventure mode + **infinite Resistance 255 + Saturation** are applied **before** the mount, so if the mount ever fails transiently the per-tick rider keeper (which recaptures adventure players) heals it a tick later; the mobs-aggro choice is applied next (`.AGGROMODE` 1 = clear any invisibility so mobs notice the rider ‚Äî also sweeps up leftovers from older pack versions or an earlier aggro-off ride; 0 = apply the infinite hidden invisibility so mobs ignore the ride from the first tick ‚Äî ¬ß6.9). (The rider's held item stays hidden because the per-tick keeper clears the inventory.) Then seeds `.s2 = .railY√ó1000`, runs `cam_follow` once ‚Äî the same absolute teleport the ride performs every tick ‚Äî to snap the whole stack (rider aboard) to its cruising position, sets `.started = 1`, and prints "Enjoy the ride." (plus a loud warning if the rider somehow still isn't seated).

**`function/setup_world.mcfunction`** (+ `overlay_snake/‚Ä¶/setup_world.mcfunction`)
One-time gamerule tuning for a clean ride: silences command feedback/output/ advancement spam; don't keep origin chunks loaded; no mob griefing (creepers/ endermen can't wreck the track); no fire spread, no phantoms; disabled tile drops; disabled all environmental damage; immediate respawn at the moving spawn point if anything impossible ever happens. It exists in **two copies** because snapshot 25w44a (format 92+, the 26.x era) renamed every gamerule to snake_case and reworked a few (`announceAdvancements` ‚Üí `show_advancement_messages`, `doInsomnia` ‚Üí `spawn_phantoms`, `doFireTick` ‚Üí removed in favor of `fire_spread_radius_around_player`, `spawnChunkRadius` ‚Üí gone). The base copy is camelCase (formats 82-91); the `overlay_snake` overlay copy is snake_case and **replaces** the base on format 92+ (see ¬ß2). `begin` calls `setup_world` once and always gets the right copy ‚Äî no dropped-file no-op, no duplicate call. Keep the two copies in sync when changing a rule. *(A full names-macro rewrite isn't worth it here: several rules aren't pure renames ‚Äî `doFireTick`‚Üí `fire_spread_radius_around_player` changes name **and** value, and `spawnChunkRadius` has no 26.x equivalent ‚Äî so two small whole files read cleaner than one macro'd file plus a big name map.)*

**`function/set_speed.mcfunction`** *(a function macro)*
A single line, `$gamerule $(rule) $(v)` ‚Äî sets the minecart max-speed gamerule named `rule` to value `v`, both read from storage `infinite_rail:speed`. **The gamerule name is a macro arg, not a literal, on purpose:** a macro line that expands to an *unknown* gamerule aborts the whole function (everything after it is skipped), so we can never afford to emit the wrong-version name. Instead the correct name is set once at load into `rule` by the version-selected `names.mcfunction`, so this line only ever runs the name valid on the running version. The rule always exists because the pack enables the **Minecart Improvements** feature in `pack.mcmeta`. Called by `begin` (with `.speed`), `speed_apply` (the Speed items' path, with the active cruise `.spcur`), `speed_up` (the ocean cruise `.ocnspd`, every ocean chunk), `speed_down` (`.speed`) and `sky_speed` (`.skyspd`).

**`function/names.mcfunction`** (+ `overlay_snake/‚Ä¶/names.mcfunction`)
Sets the version-specific command/gamerule **names** into storage: the minecart max-speed gamerule name into `infinite_rail:speed rule` (`minecartMaxSpeed` in the base copy, `max_minecart_speed` in the overlay), the weather-/daylight- cycle gamerule names into `infinite_rail:names` (`doWeatherCycle`/ `doDaylightCycle` base, `advance_weather`/`advance_time` overlay ‚Äî used by the rain/night mode toggles through the `set_rule` macro, ¬ß6.9), and the command-chain budget gamerule names (`maxCommandChainLength`/`maxCommandForkCount` base, `max_command_sequence_length`/`max_command_forks` overlay ‚Äî raised by `load`). This is the tidy home for anything that is a *pure rename* between versions: the base file holds the camelCase names, the `overlay_snake` overlay replaces it with the snake_case names on format 92+, and the shared logic reads the variable. `load` calls it once. Add more entries here as new version-renamed names come up.

**`function/set_rule.mcfunction`** *(a function macro)* `$gamerule $(rule) $(v)` ‚Äî the generic sibling of `set_speed`, reading both macro args from storage `infinite_rail:rule`. Exists for the same reason (a macro line expanding to an unknown gamerule aborts its function, so the version-correct name must come from `names.mcfunction`, never a literal). Callers copy `rule` from `infinite_rail:names` and set `v` to `"true"`/ `"false"` just before the call. Used by `mode_rain_on/off` and `mode_night_on/off` (¬ß6.9).

**`function/stop.mcfunction`**
Ends the ride: `.started=0` (which also cancels a launch still in progress), removes the `ir_rider` tag, clears effects from adventure players, takes back the pinned hotbar items (`clear` of `written_book` + `carrot_on_a_stick` ‚Äî the Ride/Visual Settings, Tips and Debug books and the Speed items), dismounts them, kills `ir_cart`, `ir_ride`, `ir_seat`, `ir_plug` and both markers, clears all forceloads. `.autodone` stays `1`, so a stopped world never auto-restarts. **The built track (blocks + `ir_disp` displays) is intentionally left in the world.**

**`function/tick.mcfunction`**
The heartbeat. Runs `menu_tick` (the menu books' `/trigger` dispatcher, ¬ß6.9) every tick, ride or no ride, so a click can never sit stale. Then, if `.started == 1`, run `main`; if `.started == 2`, run `launch_tick` (a launch is pre-building its runway ‚Äî ¬ß6.3). Below that, the **auto-starter**: while `.AUTOSTART == 1`, `.started == 0` and `.autodone ‚âÝ 1`, it waits for a player to exist, then runs a 100-tick countdown before running start, at which point `begin` sets `.autodone = 1` and it never fires again (the score persists in the world save).

**`function/main.mcfunction`**
Per-tick driver while riding:
1. Sample the pace cart's X into `.cartX`. 
1a. **Ocean speed-up:** run `ocean_check` (samples the biome once per chunk the rider enters and raises/lowers the minecart max-speed gamerule).
2. **Purity keepers:** `execute on passengers` ejects anything riding the pace cart that isn't the plug (scooped-up mobs), and anything riding the ride cart that isn't a player.
3. **Rider keeper:** any adventure player not currently riding is re-mounted ‚Äî into the ride cart normally, or directly onto the camera seat while the cart is hidden (`.HIDECART`, ¬ß6.9; a lingering ride cart is killed each tick in that mode). Handles sneak-dismounts / relog / the hide-cart toggle ‚Äî the only times the vanilla dismount hint can reappear.
4. **Pace-clear keeper:** kills every entity within 8 blocks of the pace cart except players and the ride's own entity kinds (minecarts, item/block displays, markers) ‚Äî a crowd of animals physically shoves the cart and can slow or stall it. The rider is the rig lead ahead, so the kills are never seen or heard. Java-only by design: Bedrock's pace is virtual (nothing to collide with), so it deliberately has no equivalent.
5. **Liquid keeper:** two `fill ‚Ä¶ replace` commands clear any water/lava from the pace cart's cell and the one ahead of it (each plus the cell above), so the cart never wades (water drags a minecart to a crawl) or burns. Flowing water can't waterlog the rails themselves (waterlogging only happens at placement), so clearing the open cells is the whole job.
6. **Item sweep:** kills dropped items and XP orbs within 16 blocks of the camera seat. The inventory keeper deletes pickups instantly anyway, but the pickup *sound* still played; a killed item makes none.
7. **Inventory keeper:** run `give_menu` as each adventure player ‚Äî wipes the inventory only when something beyond the seven pinned hotbar items has appeared, and re-pins a missing/wrong pinned item in place (¬ß6.9; a blanket per-tick clear + re-give used to freeze the hotbar icons on the pickup animation's first, stretched frame).
8. **Mount keepers:** unconditional `ride ‚Ä¶ mount` attempts put the plug on the pace cart and the ride cart on the seat; non-player passengers expose no vehicle NBT to query, so the attempt itself is the check (it fails silently while already seated).
9. **Stall keeper:** read `Motion[0]√ó100` into `.mx`; if `.mx ‚â§ 10` (speed < 0.1, i.e. stalled) `data merge` the pace cart's motion back to `0.5` east.
10. **Camera:** if the pace cart exists, run `cam_follow` (¬ß7g).
10b. **Minecart sound:** while `.SOUNDMODE` is 1, count the `.sndt` clock up and run `sound_loop` every 115 ticks ‚Äî the exact length of `entity.minecart.inside`, so each re-triggered `/playsound` copy starts as the previous ends. `sound_loop` only zeroes the clock when the playsound actually **reached** a rider (store-success into `.sndok`), so while the rider is still joining it simply retries every tick and the sound starts the moment they are targetable (¬ß6.9).
11. Set `.budget = .BUILD_PER_TICK` and run `build_loop` to extend the track.

**`function/ocean_check.mcfunction`**
The ocean speed-up driver, called each tick from `main` (¬ß7h). Returns immediately while `.SKYMODE` is 1 ‚Äî sky mode owns the speed, and `mode_sky_off` resets the counters and restores `.speed` on the way out. Reads the rider's X from the seat (`.rigX = ir_seat` Pos[0]) and computes its chunk `.chunkNow = .rigX / .C16`; if it equals `.lastChunk` it `return`s immediately (act only when the rider crosses a chunk boundary). Otherwise it records the new chunk, samples the biome **under the rider** (`execute at ir_seat if biome ~ ~ ~ #minecraft:is_ocean`, with the frozen oceans excluded by `unless biome` checks ‚Üí `.isOcean`) ‚Äî not the pace cart, which trails the rig lead behind ‚Äî and updates the run counters: an ocean chunk grows `.oceanRun` (and zeroes `.landRun`), a non-ocean chunk grows `.landRun` (and zeroes `.oceanRun`). While `.oceanRun ‚â• .OCEANCHUNKS` (and `.OCEANSPEED > 0`) it calls `speed_up` **each ocean chunk** (re-asserting the ocean cruise `.ocnspd` ‚Äî adjustable, default `.OCEANSPEED`); crossing `.LANDCHUNKS` non-ocean chunks while fast calls `speed_down` once. When `.DEBUGMODE == 1` it prints each chunk's biome, the running counter and the pace cart's real speed (`.dbgmx`) ‚Äî but only while the counter is still climbing to its threshold, then it goes quiet.

**`function/speed_up.mcfunction`** / **`function/speed_down.mcfunction`**
The two speed setters. `speed_up` pushes the **ocean cruise `.ocnspd`** (adjustable state, default `.OCEANSPEED` ‚Äî the Speed items tune it in both directions while the sprint is on, ¬ß6.10) through `set_speed` and is called on **every** ocean chunk past the threshold, so the cruise is continuously re-asserted and always wins over a stray `/gamerule` or a desynced state (a mid-sprint Speed click applies instantly through `speed_apply` anyway; the re-assert is the safety net); its debug line and the `.fast = 1` flip only fire on the first call (while `.fast` is still 0), so there's no spam while cruising. `speed_down` pushes `.speed` (the adjustable land speed) and is called **once**, on the transition back to land, then leaves the gamerule alone so it can still be hand-tweaked on land.

### 6.4 The build loop

**`function/build_loop.mcfunction`**
Computes `.gap = .headX ‚àí .cartX`. If there is budget left **and** the head is closer than `.PACE_CART_BEHIND` blocks to the cart **and the head marker is selectable** (loaded, entity-ticking chunk), runs `build_step`. This is the loop condition; it builds no column itself. The head gate is the **state-consistency guard**: every physical half of a column (the head `tp`, the placement, the probes) runs through `@e[tag=ir_head]` and silently no-ops when the head's chunk is unloaded, while the scoreboard half (`.railY`, `.headX`, the track history the camera flies) has no selector and would keep advancing ‚Äî the virtual line and the physical track then diverge (the camera ends up offset below/behind the real rails and the rider suffocates in uncarved ground). With the gate, a chunk-loading failure of any kind just **pauses building coherently** ‚Äî the ride runs out of track and stalls, and resumes the moment the head's chunk is back; `main` warns in chat if the head stays missing ~5 s, and `roll_chunks` warns if forceloading itself is failing.

**`function/build_step.mcfunction`**
`.budget ‚àí= 1`, `advance` (build exactly one column), then call `build_loop` again. The `build_loop`‚áÑ`build_step` recursion is a bounded loop: it keeps building columns until either the head is `.PACE_CART_BEHIND` ahead of the cart or the per-tick `.budget` is exhausted. (Recursion depth is capped by `.BUILD_PER_TICK`.)

**`function/advance.mcfunction`**
Builds **one** column (see ¬ß7 for the algorithms it drives):
1. Zero `.sum`, run `sample_window` at the head, compute `.avg = .sum / .winn` (the derived sample count).
2. `.target = .avg + .HOVER`.
2b. Run `near_scan` at the head ‚Üí `.gfloor`/`.gmax` (the ground-contact inputs for decide's slope-timing guards, ¬ß7j).
2c. Run `shift_scan` at the head ‚Üí `.sver` (the stretch shift's verified horizon, ¬ß7l).
3. `decide` ‚Üí sets `.dir` (-1/0/1) and `.veg` (this column's carve mode, ¬ß7i).
3b. If `.retro` (a slope just started): retro_clear the center bore behind the head
4. Move the head and place the column, per `.dir`:
   - `.dir 0`: `tp head ~1 ~ ~`; `place_flat`.
   - `.dir -1`: `tp head ~1 ~-1 ~`; `place_down`; `.railY ‚àí= 1`.
   - `.dir 1`: `tp head ~1 ~ ~`; `place_up`; `tp head ~ ~1 ~`; `.railY += 1`.
5. `.headX += 1`.
6. Append the (updated) `.railY` to the track-history list (the camera's map of the path; index = `.headX ‚àí .trackBase`).
6b. If `.TORCHMODE` is 1 or 2 (on or auto): run `place_torch` at the head ‚Äî torch mode's random scatter beside the new column, behind the shared `torch_auto` gate (auto plants only at night ‚Äî ¬ß6.9).
7. If `.headX ‚â• .nextLoad`, run `roll_chunks`.

### 6.5 Terrain sampling & the slope decision (the algorithm)

**`function/probe_surface.mcfunction`** / **`function/probe_down.mcfunction`** *(recursive)*
The one way any Java code reads a terrain height: snaps the `ir_probe` marker onto the **terrain** surface at the current X/Z in two passes. Pass 1 is the `positioned over motion_blocking_no_leaves` heightmap snap (ignores tree canopy and collision-less foliage, counts water/lava surfaces ‚Äî oceans read as sea level and get bridged). Pass 2 is the **dig-down**: while the block under the probe matches `#infinite_rail:not_terrain`, `probe_down` moves the probe down one block and recurses ‚Äî through tree trunks, giant mushrooms, bamboo, man-made structure blocks (village roofs, planks, glass, wool‚Ä¶) *and the air pockets under them* (house interiors, the hollow under a mushroom cap; air is in the tag), until real ground or a liquid surface is under it. Water is deliberately **not** in the tag, so liquid surfaces still count as terrain. The recursion terminates at the world floor at the latest, and a probe ending at/below Y ‚àí63 reads as a void sample (discarded by the callers). Callers (`sample_step`, `near_step`, `begin`) run it positioned at the sample column and read the probe's `Pos[1]` afterwards.

**`data/infinite_rail/tags/block/not_terrain.json`**
The `#infinite_rail:not_terrain` block tag ‚Äî everything the surface probe digs through (the terrain-height twin of the carve's `keep.json`): the whole `#infinite_rail:keep` vegetation list (nested tag reference), air (+cave/void variants, so the dig can cross interiors), snow layers, and the man-made structure families ‚Äî planks, fences/gates, stairs/slabs/walls, doors/trapdoors, glass (blocks + panes), wool + carpets, beds, campfires, cobblestone/bricks/polished-stone/cut-sandstone variants, and the village furniture blocks (hay, bell, lanterns, workstations, chests‚Ä¶). Deliberately **excluded**: water (liquid surfaces are terrain), and any block that also generates as natural ground ‚Äî sandstone, terracotta (badlands strata!), snow *blocks*, dirt paths ‚Äî ignoring those would dig whole biomes hollow. Every entry is `"required": false`, so a future rename degrades to "that block reads as terrain again" instead of breaking tag loading. Bedrock's edition of the same policy is `scripts/not_terrain.js` (¬ß11a) ‚Äî keep the pair in sync, like the vegetation pair.

**`function/sample_window.mcfunction`**
Runs positioned at the head. Computes the clamp floor `.lo = .avg ‚àí .DOWNCLAMP` (using the previous column's `.avg`; there is deliberately no upward clamp ‚Äî mountains register at full height) and the derived sample count `.winn = .SAMPLE_WINDOW / .SAMPLE_BLOCK_INTERVAL` (both knobs floored at 1 ‚Äî `.winn` is also the divisor `advance` uses, 12 at the defaults). Then walks the window as a **`sample_hop`‚áÑ`sample_step` recursion** (mcfunction has no loops; the hop distance goes through storage `infinite_rail:samp` into the `sample_hop` macro because positions only take literals): each step runs `probe_surface` (the heightmap snap + not-terrain dig-down above), reads the probe's Y into `.s`, discards void/ungenerated reads (`.s ‚â§ ‚àí63 ‚Üí .s = .avg`), floors `.s` at `.lo`, adds it to `.sum`, and hops `.SAMPLE_BLOCK_INTERVAL` further east until `.winn` samples are taken ‚Äî offsets `+4, +8, ‚Ä¶ +48` at the defaults, exactly the old fixed twelve-stanza unroll this replaced. **The down-clamp is what makes narrow ravines barely move the average** (so they get bridged level) while broad valleys still lower it.

**`function/near_scan.mcfunction`** / **`function/near_step.mcfunction`** *(recursive)*
The near-ground scan (¬ß7j), run at the head between the sample window and `decide`. `near_scan` takes the scan reach `.nw = .SAMPLE_WINDOW` (floored at 1 ‚Äî the scan never reads past the sampling horizon, and `.DOWNLOOK_AHEAD` is thereby capped by it), seeds the accumulators, and starts `near_step` positioned one block east. Each `near_step` snaps the `ir_probe` marker onto the surface (the same `probe_surface` used by `sample_window`, so trees and structures are already invisible), folds the read and its predecessor into a **pair** (`.pmin` = min of the two ‚Äî narrow spikes of real terrain vanish, ¬ß7j), accumulates `.gfloor` (max pair within `.DOWNLOOK_AHEAD`), `.gmax` (max pair anywhere in the walk ‚Äî the climb side always uses the full window) and `.gcone` (max 45¬∞-projection `.pmin ‚àí distance` over pairs above `.railY ‚àí .HOVER`), then hops 2 blocks east and recurses ‚Äî so the probes land at odd offsets `~1, ~3, ~5, ‚Ä¶`. Void/ungenerated reads break the pair chain and are skipped; if the scan got no valid probe at all (`.gnu` = 0), `.gcone` falls back to the `+32000` never-hold sentinel.

**`function/shift_scan.mcfunction`** / **`function/shift_step.mcfunction`** *(recursive)*
The stretch-shift scan (¬ß7l), run at the head between `near_scan` and `decide`. `shift_scan` presets `.sver = 0` and returns early unless the shift question is live (`.GAPSTRETCH ‚â• 1`, running flat, a descent of at least `.MIN_CHANGE` wanted, the flat run not already past both gaps, and depth + `.GAPSTRETCH` within the 96-block reach); otherwise it seeds the walk and `shift_step` probes every 2 blocks (odd offsets, `probe_surface`, paired mins like the near scan) verifying each pair against the planned shifted profile: ground must stay `.DOWNGRACE` below `railY ‚àí min(offset, depth)` everywhere, and beyond the descent must also sit **at** the landing level (‚â• `railY ‚àí depth ‚àí .HOVER ‚àí .MIN_CHANGE`). The first violation or void read ends the walk; `.sver` keeps the horizon verified so far.

**`function/decide.mcfunction`**
Chooses this column's `.dir` using the **event model** (¬ß7b). Computes `.diff = .target ‚àí .railY`, snapshots `.slope0 = .slope`, and derives the four ground-contact guard flags from the near scan (¬ß7j): `.dig` / `.dig2` (one / two more down-steps would land the rail below the descent floor `.gfloor + .DOWNGRACE`), `.push` (the rail is not yet a full `.HOVER` above the highest ground the scan sees, and still under `.target + .UPGRACE`) and `.due` (the climb schedule allows starting: the rail is within `.UPEARLY` of the cone's demanded height `.gcone + .HOVER`). All stay inert while `.SKYMODE` is 1 (sky mode holds `.SKYY` dead level and punches through whatever it meets); the descent side additionally disables at `.DOWNLOOK_AHEAD` 0 (the climb side is always on ‚Äî its no-data sentinels fail open).
- If an event is in progress (`.slope0 = ¬±1`): keep sloping the same way until the rail reaches the target ‚Äî climb while `.diff ‚â• 1` **or `.push` is 1** (crest completion: finish at hover height over ground the level line would still hit, up to `.UPGRACE` past the target); descend while `.diff ‚â§ ‚àí1` **and `.dig` is 0** ‚Äî a blocked descent ends early (`end_event`), resting just above the ground, and the line continues downward as a new, gap-paced event once the ground falls away.
- If flat (`.slope0 = 0`): call `consider_start` to maybe begin a new event.

**`function/consider_start.mcfunction`**
Decides, when flat, whether to begin a climb/descent:
- `.want = 1` if `.diff ‚â• .MIN_CHANGE`; `.want = ‚àí1` if `.diff ‚â§ ‚àí.MIN_CHANGE` (via `.ndead = ‚àí.MIN_CHANGE`); else `0`.
- **Ground-contact overrides (¬ß7j):** a wanted climb is *held* (`.want 1 ‚Üí 0`) while the schedule says it is not yet due (`.due` 0 ‚Äî the 45¬∞ cone still has more than `.UPEARLY` blocks of headroom; `.flat` keeps counting during the hold). A climb is also wanted inside the deadband (`.want 0 ‚Üí 1`) when `.diff ‚â• 1`, `.gmax > .railY` (the level line is about to plow into rising terrain) and the schedule agrees. A wanted descent is vetoed (`.want ‚àí1 ‚Üí 0`) while `.dig2` is 1 ‚Äî never start a descent without clear runway for at least two steps; hold level and let the ground fall away first.
- If `.want = 0`: stay flat, `.flat += 1` (count toward the next gap).
- If `.want ‚âÝ 0`: pick `.need = .SAMEGAP` (if `.want == .lastDir`) or `.TURNGAP` (reversal), then subtract the **big-event gap credit** (¬ß7k): `.evrun √ó .GAPRATIO / 100` (the knobs are percents ‚Äî √ó100 fixed point via `.C100`), provided the wanted move is itself at least `.evrun √ó .GAPMATCH / 100` blocks (for climbs the move's size is read from the near scan ‚Äî `.gmax + .HOVER ‚àí .railY` ‚Äî where it is larger than the diluted average's `.diff`); the result is floored at 0. Then the **stretch shift** (¬ß7l): a wanted descent whose verified horizon `.sver` covers `|.diff| + .GAPSTRETCH` ‚Äî and which is itself at least `.need √ó .GAPMATCH / 100` tall ‚Äî sets `.need` to 0 outright. If `.flat ‚â• .need`, call `start_event`; otherwise **hold level** (`.flat += 1`, guarded by `.slope == 0`). Holding is what produces bridges (the ground drops away under a level rail) and tunnels (the ground rises into it).

**`function/start_event.mcfunction`**
Begins an event: `.dir = .want`, `.slope = .want`, `.lastDir = .want`, `.flat = 0`. This column becomes the first sloped column; `decide` continues the slope on subsequent columns until the target is reached.

**`function/end_event.mcfunction`**
Ends an event: `.slope = 0`, `.flat = 0`. `.dir` stays `0`, so the current column is placed flat at the elevation just reached, and gap-counting restarts.

### 6.6 Column geometry (how slopes map to blocks)

All three run positioned at the head; the head is already at this column's `(X, railY, Z)`. **Order matters:** the carve happens first, then `support` (which lays the redstone block *under* the rail), then the rail, then the light ‚Äî because the track hovers above the ground, so the cell under the rail is air and the rail would pop off if placed before its support existed. The carve height is configurable (`.TUNNELCLEAR`), and the carve is **vegetation-sparing** (¬ß7i).

**`function/place_flat.mcfunction`**
Sets the carve height (`.TUNNELCLEAR`) into both the `.ch` score (the per-cell walk) and storage `infinite_rail:carve h` (the full-clear fill macro), runs `carve` (3 wide √ó `.TUNNELCLEAR+1` cells tall ‚Äî the rail cell plus `.TUNNELCLEAR` above); `support`; `powered_rail[shape=east_west,powered=true]` at `~`; `place_light` (the track light at `~3`, level per the Track light mode ‚Äî below).

**`function/place_light.mcfunction`** / **`function/light_at.mcfunction`** *(a function macro)*
The track light above the rail, shared by all three place functions: `place_light` reads the Track light mode `.LIGHTMODE` (¬ß6.9) ‚Äî `0` (off) places nothing at all (the carve already left the cell clear); `1..15` copies the level into storage `infinite_rail:light l` and calls `light_at`, whose one macro line `$setblock ~ ~3 ~ minecraft:light[level=$(l)]` places it (block states can't come from a scoreboard). Presets via `mode_light_on/low/off`: 11 bright (default), 8 dim, 0 off. New columns only ‚Äî built track keeps whatever it got. A hand-set level above 15 simply aborts `light_at`'s macro line (no light that column), nothing worse.

**`function/place_up.mcfunction`**
Climbing column. Same as flat but carves with `.TUNNELUP` (= `.TUNNELCLEAR+1`, one block of extra headroom as the cart rises) and places `powered_rail[shape=ascending_east,powered=true]`. (Slope columns always full-clear their center bore: `decide` sets `.veg` 0 on them.)

**`function/place_down.mcfunction`**
Descending column. Carves with `.TUNNELUP`; places `powered_rail[shape=ascending_west,powered=true]`. (Because a descent moves the head down first, the rail sits one lower and slopes up toward the west behind it, which is the same physical staircase as a climb viewed the other way.)

**`function/carve.mcfunction`**
The vegetation-sparing clearance bore (¬ß7i), positioned at the head. Always clears the **critical envelope** ‚Äî the rail cell and the cell above it, center only ‚Äî with one literal `fill`. If `.veg` is 0 (slope / slope-buffer column) it clears the rest of the center bore in one `carve_center` fill; then it walks the bore per-cell with `carve_layer` (`.cy` 0 ‚Üí `.ch`).

**`function/carve_center.mcfunction`** *(a function macro)*
`$fill ~ ~2 ~ ~ ~$(h) ~ minecraft:air` ‚Äî the full center clear above the envelope for `.veg 0` columns. `fill` needs literal coordinates, so the height arrives as a macro arg (storage `infinite_rail:carve h`, set by the `place_*` caller to `.TUNNELCLEAR` or `.TUNNELUP`).

**`function/carve_layer.mcfunction`** *(recursive)*
One horizontal slice per call, climbing `positioned ~ ~1 ~` from the rail cell to `.ch` above it. Each cell is set to air **unless** it matches the `#infinite_rail:keep` block tag (Java's vegetation list, `tags/block/keep.json`): the two side cells always get the sparing test, the center cell only in veg mode (`.veg` 1) and only from 2 above the rail up (the envelope below was already cleared).

**`function/retro_clear.mcfunction`**
Runs from `advance` when the shared `start_event` raises `.retro` (a slope begins this column): computes the span `.rk = min(.SLOPECLEAR, columns built this ride)` ‚Äî so it can never reach behind the start point ‚Äî stores `k`/`h` and delegates to `retro_fill`. Positioned at the head, which still sits on the last **built** column.

**`function/retro_fill.mcfunction`** *(a function macro)*
`$fill ~-$(k) ~2 ~ ~ ~$(h) ~ minecraft:air` ‚Äî the retroactive full-height center clear over the flat columns just before a slope (they were carved vegetation-sparing, but the camera lifts off the rail line early ‚Äî ¬ß7g). Vertical only: the side cells keep their plants.

**`data/infinite_rail/tags/block/keep.json`**
The `#infinite_rail:keep` block tag ‚Äî everything the carve spares: Java's edition of the vegetation list, maintained by hand in `src/java/` (Bedrock keeps its own edition in `src/bedrock/bp/scripts/vegetation.js` ‚Äî the two are independent files because the editions' block naming differs anyway; keep them in policy sync when adding a plant, ¬ß11a). Every individual block-id entry is `"required": false`, so a future block rename degrades to "that plant gets carved again" instead of breaking tag loading.

**`function/support.mcfunction`**
Lays the power+disguise under the rail (shared by all three place functions):
- `setblock ~ ~-1 ~ minecraft:redstone_block` ‚Äî a block of redstone directly under the rail. It **powers the powered rail resting on it**, is **immune to water**, and **emits no light** (so it can't wash away or melt ice). This single block replaces the old 5-block stone/torch/stone stack + barriers.
- `execute align xyz run summon minecraft:block_display ‚Ä¶` ‚Äî a smooth-stone `block_display` (tag `ir_disp`) that disguises the red block. Details that matter:
  - `align xyz` snaps the summon to the block corner (the head is block-centered).
  - `brightness:{sky:15,block:15}` is **required** ‚Äî a display samples the light of the cell it occupies, which contains the opaque redstone block (light 0), so without the override it renders solid black.
  - `scale:[1, 1.01, 1.01]` / `translation:[0, ‚àí0.005, ‚àí0.005]` ‚Äî enlarged a hair in **Y and Z only** so the visible faces (underside + the two sides seen from a bridge) sit just outside the redstone block and don't z-fight it. X stays exactly 1 so neighboring supports (one block apart along the track) touch but never overlap ‚Äî a uniform >1 scale made adjacent displays overlap and shimmer.

### 6.7 Chunk management

**`function/roll_chunks.mcfunction`**
Runs every 16 blocks of head travel (gated by `.nextLoad` in `advance`), positioned at the head. Presets `.flok = 0` and runs `forceload_here` (which computes the corridor arguments and calls the `forceload` macro ‚Äî generate ahead, release behind ‚Äî storing its **success** back into `.flok`); if `.flok` is still 0 ‚Äî the file failed to load on this game version, the macro expansion aborted, or the command errored ‚Äî it prints a **one-shot chat warning** ("chunk force-loading is failing‚Ä¶", re-armed after the next success) instead of letting the ride starve silently. Then `setworldspawn` and `spawnpoint @a` at `~ ~1 ~` so world spawn and the player's respawn point **roll forward with the ride** (nothing anchors to the origin); `.nextLoad += 16`.

**`function/forceload_here.mcfunction`**
Computes the `forceload` macro's two arguments into storage `infinite_rail:args` and calls it at the current position (the head for `roll_chunks`, the starting player for `begin`): `gen` = `.TERRAIN_GENAHEAD`, and `w` = the corridor's Z half-width ‚Äî 8 (¬±1 chunk) normally, raised to `.TORCHRANGE` (capped 48) **while torch mode is on or auto** (`.TORCHMODE` 1..), so torches thrown past the standard band still land in loaded, generated chunks instead of silently failing to place. **Deliberately clock-blind**: the width keys off the mode score alone, never off torch_auto's live is-it-night answer ‚Äî this file is the heart of the chunk pipeline, and `/time` is the pack's most version-turbulent command (26.1 reworked it around World Clocks); anything that fails to parse here kills forceloading outright, and a dead corridor is what let the scoreboard state and the physical track diverge. A few extra daytime chunks in auto mode are cheap insurance by comparison; the clock read lives quarantined in `time_now`, called only by `place_torch`.

**`function/time_now.mcfunction`** + **`predicate/night.json`** / **`predicate/day.json`**
The **one** place Java answers "is it night right now?" ‚Äî via the **`#infinite_rail:night` predicate** (vanilla's `minecraft:time_check`, period 24000, window 12542..23459), evaluated fresh on every `execute if predicate` against the same day-time value that renders the sky ‚Äî so it is correct no matter how the current time came about (natural rollover, `/time set`, a frozen cycle). The predicate is **version-split like `setup_world`**: 26.1's World Clock rework made a `clock` field **required** on `time_check`, so the base copies carry the classic 1.21 form and `overlay_snake` (format 92+) replaces them with copies naming `"clock": "minecraft:overworld"` ‚Äî the vanilla Overworld clock whose day/noon/night/midnight markers `/time set` drives. (A copy that doesn't decode on some version just never registers: `check_clock`/`time_now` then fail to compile in isolation, load's warning fires, and auto degrades to unlit ‚Äî on the handful of dead mid-2025 snapshot formats between 92 and the World Clock's debut, that warning is the expected state.) The night window now lives in FOUR places to keep in sync: both predicate copies, `torch_auto`, and Bedrock's `torchLit()` fallback. The answer is handed onward as a *representative* day time (`.tod` = 18000 in-window / 6000 outside) so the shared `torch_auto` stays byte-identical across editions (Bedrock feeds it a real 0..23999 reading). **Why not `time query daytime`:** that was the original fetch and it broke on 26.2 ‚Äî the 26.1 World Clock rework changed what /time's stored query results report and the reading stopped tracking `/time set`, so auto mode stayed dark even in a forced night. Still quarantined in its own file (the post-mortem rule: clock-adjacent code must never be able to take another file down with it ‚Äî `execute if predicate` itself refused to compile inside `load.mcfunction` on 26.2 and killed the entire bootstrap, so even the *self-test's* predicate probes live in their own quarantined `check_clock.mcfunction`); a broken/unloaded predicate evaluates false = never night, degrading auto to unlit and nothing worse ‚Äî `load` presets `.todok` 0, calls `check_clock` (which flips it to 1 when either the `night` or `day` predicate matches ‚Äî `day.json` is `minecraft:inverted` around the same check, so exactly one must match at any moment), and warns loudly if it stayed 0, whether because the predicates read false or because `check_clock` itself wouldn't load. Keep the predicate's window in sync with `torch_auto`'s.

**`function/forceload.mcfunction`** *(a function macro)*
`forceload` only accepts literal/relative coordinates, not scoreboard values, so both distances arrive as macro args:
- `forceload remove ~-336 ~-64 ~-256 ~64` ‚Äî release a band well behind the head; as the head advances 16 at a time these bands tile to clear everything ‚â≥256 blocks back. The ¬±64 half-width is fixed and generous on purpose: it covers every width the add line can have used (releasing a never-forced chunk is a no-op), so lowering `.TORCHRANGE` mid-ride can't strand wide chunks loaded behind the ride. Runs at the caller's position (head), inherited via the call.
- `$return run forceload add ~ ~-$(w) ~$(gen) ~$(w)` ‚Äî force-generate the corridor from the head out to `.TERRAIN_GENAHEAD` blocks ahead, `$(w)` blocks to each side.

**Order and the explicit `return run` are load-bearing:** `forceload_here` store-successes this whole function into `.flok`, the health signal `roll_chunks` warns on ‚Äî and on modern versions a function *without* an explicit `/return` stores success **0** no matter what its commands did, so the signal read "failing" (and fired its one-shot warning at every ride start) even while forceloading worked. The remove runs first ‚Äî its failure is routine, since for the first ~336 blocks of a ride there is nothing behind to release ‚Äî and the function returns the **add's** own result, the one command whose success actually means the corridor ahead is loading (the add gains at least one brand-new chunk column per 16-block roll, so a healthy pipeline always answers 1).

### 6.8 Smooth camera (the ride rig)

**`function/cam_follow.mcfunction`**
The per-tick camera driver, called from `main` (gated on `ir_cart` existing; returns immediately if there is no track history, e.g. the pack was updated over a ride in progress). Reads the pace cart's X once as fixed-point (`.cxm = X√ó1000`) and derives both the sub-block fraction `.fx` (floorMod) and the rig's column index `.ci` (cart column + the rig lead, clamped to the valid history range) from it; precomputes `.lift`/`.wmax`/`.half`; reads the rail line at the rig (`.linem`, one `cam_sample`); computes the two candidate heights ‚Äî `.c1`, the constructed S-curve (blend loop `cam_blend`), and `.s2 += (.linem ‚àí .s2)/.CAMSMOOTH`, the reactive descent chaser ‚Äî and takes `.sy = max(.c1, .s2)`, floored at `.linem`; then `cam_move`. See ¬ß7g.

**`function/cam_blend.mcfunction`** *(recursive)*
One S-curve sample per call: offset `.j` runs from ‚àí`.CAMBLEND/2` to +`.CAMBLEND/2` in steps of 1. Each sample computes `lifted(j) = min(max of the profile over [j .. j+.wmax+1], line(j) + .lift)` via `cam_scan`, and accumulates `.tsum`/`.tn`; `.c1` is their average ‚Äî an average over a symmetric window reproduces straight stretches exactly and turns every corner of `lifted()` into a parabolic blend `.CAMBLEND` long.

**`function/cam_scan.mcfunction`** *(recursive)*
The small forward-max scan for one blend sample: `.k` runs 0 to `.wmax` in steps of 1, tracking the highest interpolated height `.fmx` and capturing the k = 0 sample as `.l0`. Scanning further than `.CAMLIFT`+2 blocks is pointless (the `+.lift` cap clips anything higher), which is also what keeps lift-off from starting any earlier than the blend needs.

**`function/cam_sample.mcfunction`**
Reads one interpolated profile height into `.sm`: column `.si` (clamped to the built range) and its neighbor, blended by `.fx`/`.fi` so values move continuously as the cart crosses block edges.

**`function/cam_get.mcfunction`** *(a function macro)*
`$execute store result score .ly ir run data get storage infinite_rail:track y[$(i)]` ‚Äî NBT paths only take literal indices, so the index arrives as a macro arg (storage `infinite_rail:cami i`).

**`function/cam_move.mcfunction`**
Teleports the seat ‚Äî and with it the rigid ride-cart + rider stack ‚Äî to the rig lead (`.PACE_CART_BEHIND ‚àí .RIDER_BEHIND`) blocks east of the pace cart at height `.sy + 62 + .CAMHEIGHT√ó100` milli (62 ‚âà how high a minecart rests above a rail, so the ride cart sits on the smoothed line like a real cart). Runs `cam_tp` **positioned at the pace cart**, so X/Z are relative offsets and never pass through a scoreboard (full double precision forever).

**`function/cam_tp.mcfunction`** *(a function macro)*
One line: `$tp @e[type=item_display,tag=ir_seat,limit=1] ~$(dx) $(y) ~` ‚Äî relative X (the rig-lead offset) and Z with an absolute Y. `tp` only takes literal/relative coordinates, so the values arrive as macro arguments from storage `infinite_rail:cam`.

### 6.9 Ride modes

Optional flavors toggled by chat command (`/function infinite_rail:mode_*`) ‚Äî or through the **Settings books**, the two in-game menu items pinned into the rider's hotbar (`give_menu`, ¬ß6.10): **Ride Settings** (how the ride moves, sounds and looks from the seat: sky mode, the cart sound, hide-cart, mobs aggro) and **Visual Settings** (what the world looks like: rain, time, torches, the track light ‚Äî page 2). Rain, sky, hide-cart, sound and mobs-aggro are `_on`/`_off` pairs; time, torches and the track light are **tri-states** (time: night only / day only / default ‚Äî the three-way `.NIGHTMODE`; torches: off / always on / **auto** ‚Äî night-only planting, the default ‚Äî the three-way `.TORCHMODE`; track light: off / low / on ‚Äî `.LIGHTMODE` 0/8/11, the light level itself). They are **independent switches, not a mutually exclusive mode select** ‚Äî any combination stacks ‚Äî and they are **state, not config**: the toggles live in the `.RAINMODE`/`.NIGHTMODE`/`.TORCHMODE`/ `.SKYMODE`/`.HIDECART`/`.SOUNDMODE`/`.AGGROMODE`/`.LIGHTMODE` scores (¬ß4.1), seeded by the shared `modes_init` and untouched by `config`, so a `/reload`, a ride restart, `stop`, or a rejoin never turns a mode off. The knobs shaping them (`.SKYY`, `.SKYSPEED`, `.TORCHRANGE`, `.SEAPICKLE`) are ordinary config tunables; the torch **density** is menu-driven state (`.torchdens`, the Low/Medium/High/Max presets below) with `.TORCHODDS` as its config-side default, and the sound toggle likewise seeds from a config default (`.CARTSOUND`). `.SEAPICKLE` (0-4) is torch mode's over-water fallback ‚Äî a sea pickle on the bed where a torch would land on water ‚Äî with no in-game toggle of its own.

**`modes_init.mcfunction`** *(shared source: `src/shared/functions/`)*
Seeds the four zero-default toggle scores with `scoreboard players add ‚Ä¶ 0` ‚Äî creates a missing score at 0, leaves a set one alone ‚Äî and seeds the **four** adjustable state values the same way: `.speed` copies `.DEFAULTSPEED`, `.skyspd` (the adjustable sky cruise) copies `.SKYSPEED`, `.ocnspd` (the adjustable ocean cruise) copies `.OCEANSPEED`, and `.torchdens` copies `.TORCHODDS`, each only when unset or invalid, so a chosen land/sky/ocean speed or density persists (¬ß6.10/¬ß6.9). Four modes have **non-zero defaults**, which add-0 can't produce, so each gets a one-shot companion flag (add-0 then checked-and-set) gating the seed to the first load only: `.TORCHMODE` starts at 2 = auto (`.trchinit` ‚Äî the seed additionally requires the score to be 0, so a world upgrading from the on/off era keeps a deliberately switched-on 1), `.AGGROMODE` starts at 1 = mobs aggro (`.agginit`), `.LIGHTMODE` starts at 11 = the bright track light (`.lgtinit` ‚Äî 0/off is a legitimate choice, indistinguishable from "never set" without the flag), and `.SOUNDMODE` copies its config default `.CARTSOUND` (`.sndinit` ‚Äî for a 0/1 toggle "never set" and "off" are the same number). Called from `load` (Java) and the script's `init()` (Bedrock), always after `config`.

**`function/mode_rain_on.mcfunction`** / **`mode_rain_off.mcfunction`**
Permanent rain. `_on` freezes the vanilla weather cycle (the version-correct gamerule via `names` ‚Üí `set_rule`) and runs `weather rain` ‚Äî with the cycle frozen the rain can never time out. `_off` re-enables the cycle and clears the sky. Pure world state: works with or without a ride running.

**`function/mode_night_on.mcfunction`** / **`mode_day_on.mcfunction`** / **`mode_night_off.mcfunction`** (+ `mode_day_off`, an alias for `mode_night_off`)
The **tri-state time mode** (`.NIGHTMODE` 0/1/2 ‚Äî ¬ß4.1): `mode_night_on` = night only (cycle frozen + `time set midnight`, moon at its peak), `mode_day_on` = day only (cycle frozen + `time set noon`, sun at its peak), `mode_night_off` = back to the default (cycle re-enabled, morning). Same version-picked gamerule plumbing as rain.

**`function/mode_torches_on.mcfunction`** / **`mode_torches_auto.mcfunction`** / **`mode_torches_off.mcfunction`**
Set the **tri-state `.TORCHMODE`** (1 / 2 / 0): always on, **auto** (torches beside new track only while it is night ‚Äî the default), or off. The placement itself is `place_torch`/`torch_try`, hooked into `advance` (step 6b); auto's is-it-night decision is the shared `torch_auto` (below). Auto composes with the time modes for free: `mode_night_on` freezes the clock inside the night window (torches around the clock), `mode_day_on` outside it (never).

**`torch_auto.mcfunction`** *(shared source: `src/shared/functions/`)*
Torch mode's placement gate ‚Äî the one place both editions decide whether new columns get torches *right now*. Inputs: `.TORCHMODE` and `.tod` (the world clock, fetched natively by each caller just before the call ‚Äî Bedrock has no `execute store`, so Java runs the quarantined `time_now` and Bedrock hands `world.getTimeOfDay()` in through the brain bridge). The clock is **floor-modded by `.C24000` first** (a 26.1-era world clock may report total elapsed ticks rather than a 0..23999 day time; the modulo makes every variant read alike, and an unset `.tod` ‚Äî a broken clock fetch ‚Äî reads as 0 = unlit). Output: `.torchlit` ‚Äî 1 in mode 1 always, and in mode 2 only inside the **night window 12542..23459** (the vanilla beds-usable span, sun half-set at dusk to first light of dawn). Callers: Java `place_torch` (per candidate column) and Bedrock `torchLit()` (cached ~1 s; the corridor widening in `forceload_here` is deliberately clock-blind ‚Äî ¬ß6.7). Nothing is ever removed at daybreak ‚Äî torches simply stop appearing on new columns.

**`function/torch_density_low.mcfunction`** / **`torch_density_medium`** / **`torch_density_high`** / **`torch_density_max`**
The Visual Settings menu's torch-density presets: each sets the `.torchdens` state score (Low 15, Medium 35 ‚Äî the default, High 70, Max 100) and prints the **friendly name only** ‚Äî the percentage is deliberately never shown to the user, in chat or the menus. `.torchdens` is what the per-column roll actually reads (Java `place_torch`, Bedrock `maybeTorch()`); it is seeded from config `.TORCHODDS` by `modes_init` and, being state, a chosen density survives reloads, rejoins and ride restarts. Both editions carry the same four files (Java tellraw vs Bedrock rawtext ‚Äî keep the values in sync); Bedrock's Visual Settings form drives them through a dropdown, Java's Visual Settings book through `ir_menu` 22-25.

**`function/place_torch.mcfunction`**
Runs positioned at the head, once per built column while `.TORCHMODE` isn't off (1..). First the gate: fetch the clock into `.tod` (via the quarantined `time_now` ‚Äî ¬ß6.7) and run the shared `torch_auto` ‚Äî a daytime column in auto mode returns right here (`.torchlit` 0). Then roll one (`random value 1..100` vs `.torchdens`, the density state score behind the menu presets): does this column get a torch at all? Roll two: how far out ‚Äî `/random` can only roll literal ranges, so a fixed `0..99` roll is scaled in fixed point to a uniform 2..`.TORCHRANGE` distance (clamped 2‚Äì48; the floor of 2 stays clear of the 3-wide carve). Roll three (`0..1`): which side ‚Äî the result is written to storage `infinite_rail:torch dz` with store scale **+1 or ‚àí1**, folding distance and side into one signed offset for the macro hop.

**`function/torch_at.mcfunction`** *(a function macro)*
`$execute positioned ~ ~ ~$(dz) run function infinite_rail:torch_try` ‚Äî positions can't come from scoreboards, so the signed Z offset arrives as a macro arg. Runs at the head (inherited from `place_torch`'s caller).

**`function/torch_try.mcfunction`**
Plants one torch at the rolled X/Z: `positioned over motion_blocking_no_leaves` snaps to the surface (under forest canopy, not on it), then attempts the placement on whatever ground is there. **Water gets a sea pickle instead** (below); lava or a lily pad is still skipped (a torch over liquid floats then pops). Everything else gets its torch attempt, so frozen/snowy biomes are lit too: ice of all kinds holds a torch, and a snow **layer** occupying the target cell is *replaced* by the torch (its own `if block ‚Ä¶ snow` line ‚Äî what hand-placing on snowy ground does; `setblock ‚Ä¶ keep` would silently no-op on it, which is why snowfields used to go torchless). Elsewhere `keep` only fills air, so occupied cells stay silent no-ops. (A torch on regular ice can melt it on a later random tick and pop ‚Äî no drop, `doTileDrops` is off; accepted trade for lighting frozen lakes at all.)
The **water case** (config `.SEAPICKLE`, 1‚Äì4 = pickles = brightness, 0 = the old skip): when the surface below really is water, re-snap with `positioned over ocean_floor` ‚Äî the same heightmap idea as `motion_blocking_no_leaves` but it *also* excludes fluids, so it drops onto the true sea/lake/river bed ‚Äî and, if that bed cell is water, plant a waterlogged `sea_pickle[pickles=N]` there via the `pickle_place` macro (the count arrives through storage `infinite_rail:pickle n` because block states can't be read from a scoreboard). A torch is light 14; 4 pickles is light 15, the closest match. Always on by default; there is no in-game Settings toggle for it.

**`function/pickle_place.mcfunction`** *(a function macro)*
`$setblock ~ ~ ~ minecraft:sea_pickle[pickles=$(n),waterlogged=true]` ‚Äî plants `.SEAPICKLE` pickles in the bottom water cell (positioned there by `torch_try`), waterlogged so they stay submerged and glow. No `keep`: the caller already verified the cell is water, and the waterlogged pickle keeps the water visual.

**`function/mode_light_on.mcfunction`** / **`mode_light_low.mcfunction`** / **`mode_light_off.mcfunction`**
The **Track light** tri-state: set `.LIGHTMODE` to the light level placed 3 above every **new** rail ‚Äî 11 (on ‚Äî the bright classic line, the default and the ice-melt-safe maximum), 8 (low ‚Äî a dim glow, subtler at night but still spawn-proof on the line), or 0 (off ‚Äî no light block at all: tunnels bore dark, nights stay dark, and hostile mobs can spawn in the unlit tunnels the ride leaves behind). The placement itself is Java `place_light`/`light_at` (¬ß6.6) and Bedrock `placeColumn` (`lightLevel()` ‚Üí `light_block_<n>`); like torch mode it affects **new columns only** ‚Äî the already-built line keeps its lights. Reached from the Visual Settings menus (Java book page 2, Bedrock's dropdown) or by chat.

**`function/mode_hidecart_on.mcfunction`** / **`mode_hidecart_off.mcfunction`**
Hide/show the minecart (`.HIDECART`). On **Java** the visible cart is a real vehicle, so hiding means changing the rig: `_on` kills the ride cart (dismounting the rider with it) and the next tick's rider keeper re-seats them **directly on the invisible camera seat** ‚Äî same rigid stack, one piece shorter; `_off` re-summons the cart at the seat (mid-ride, mirroring `launch_done`'s summon line), perches it back and unseats the rider so the keeper moves them into it. `launch_done` and the keeper both branch on the score, so the mode works at ride start and mid-ride, and each toggle costs exactly one vanilla dismount-hint toast (unavoidable ‚Äî it accompanies every player mount). Sitting on the bare seat reads a hair different from cart-passenger height; `.CAMHEIGHT` can fine-tune if desired. On **Bedrock** the cart prop is pure scenery and the rider already sits on the seat, so the files only flip the score ‚Äî `camMove` just glides the prop at the fixed `HIDE_CARTYOFF` sink (‚àí0.5 blocks, replacing `.CARTYOFF`) below the track line while the mode is on, and resumes the live-read config `.CARTYOFF` when it turns off; nothing is despawned and toggling is completely seamless there. (The Ride Settings form also verifies the score actually flipped and writes it directly if the `mode_hidecart_*` function call didn't take ‚Äî a same-version pack update can leave Bedrock's function registry stale, which is also why the manifests' versions must be bumped with every release.)

**`function/mode_sky_on.mcfunction`** / **`mode_sky_off.mcfunction`** / **`function/sky_speed.mcfunction`**
The high-altitude cruise. The elevation half lives in the **shared `decide`**: while `.SKYMODE` is 1 the terrain-derived `.target` is replaced with `.SKYY` before any slope decision, so the ordinary event model climbs to it in one contiguous 45¬∞ event, holds it dead level (the target never moves), and glides back down when the mode ends ‚Äî no new machinery, just a different opinion about where the rail wants to be. The terrain sampler keeps running underneath, so the descent lands correctly wherever the ride happens to be. The speed half: the ride cruises at the **adjustable sky speed `.skyspd`** (seeded from `.SKYSPEED`) while the mode is on ‚Äî `_on` applies it via `sky_speed` (which `begin` also re-applies if a ride starts while the mode is on), Bedrock's `tickPace` asserts it every tick ‚Äî and the Speed ‚àí/+/Reset items **tune `.skyspd` while sky mode is on** (the shared `speed_step` targets it when `.SKYMODE` is 1), so sky mode jumps to its default on entry but is adjustable from there; the land speed `.speed` is left untouched. `ocean_check` returns early while the mode is on; `_off` restores the land speed `.speed` (Java) / hands the speed back to the ocean system (Bedrock's `tickPace` on the toggle-off transition) and zeroes `.fast`/`.oceanRun`/`.landRun` so the ocean system resumes fresh. Terrain above `.SKYY` is punched through like any rise the rail can't out-climb.

**`function/mode_sound_on.mcfunction`** / **`mode_sound_off.mcfunction`** / **`sound_loop.mcfunction`**
The minecart sound (Java side; Bedrock's is ¬ß11a/¬ß11e). The ride cart never rolls on rails ‚Äî it is a passenger of the camera seat, and the client zeroes every passenger's velocity each tick (`rideTick`), so the engine never plays its in-cart loop ‚Äî and the pace cart that *does* roll trails the rig lead (~64 blocks) back, past the vanilla 16-block earshot. So the sound is faked: while `.SOUNDMODE` is 1, `main` counts the `.sndt` clock and every **115 ticks** runs `sound_loop`, which plays vanilla's first-person riding sample `entity.minecart.inside` at the rider ‚Äî 115 ticks because that sample (`minecart/inside.ogg`) is 5.77 s = 115.4 ticks long, so each played copy starts just as the previous one ends, reading as one continuous loop out of a sound the data pack cannot itself loop.
The trick that makes it sound right (an earlier attempt faded and gapped): a `/playsound` is emitted at a **fixed** world point, and the ride glides away from it ‚Äî up to ~185 blocks over one 5.77 s copy on an ocean sprint. The fix is the **volume argument**. On Java a volume above 1.0 does *not* make a sound louder ‚Äî it only extends the distance it carries (the listener's perceived loudness is still capped at full). At **100** the audible radius is enormous, so the rider sits deep in the flat-volume zone for the whole copy no matter how far they travel: constant volume, no fade. Only the rider is a target (`@a[tag=ir_rider]` ‚Üí `@s`), so the large volume is never heard by anyone else. It's a fixed cadence ‚Äî unlike a real cart's, the loop doesn't pitch or swell with speed. `entity.minecart.inside` is the *first-person* riding sample (what you hear sitting inside a cart), reachable because `/playsound` takes any registered sound event by name ‚Äî no cart, no resource pack.
`mode_sound_on` primes `.sndt` at the threshold (instant start; `launch_done` does the same at ride start, and **`load` too** ‚Äî a world rejoin resumes the persisted clock wherever it was mid-count, which used to leave the ride silent for up to 5.75 s after loading in; a mid-ride `/reload` therefore restarts the sample once, a single clean seam), `mode_sound_off` and `stop` run `stopsound` to cut the up-to-5.8-second tail of the copy in progress. `sound_loop` zeroes the clock **only when the playsound actually reached a rider** (`execute store success` ‚Üí `.sndok`): while the rider's player entity doesn't exist yet (world load resumes a ride before they finish joining) the selector matches nobody, and unconditionally resetting the clock used to swallow that whole cycle ‚Äî with the guard it retries every tick until the rider is targetable. **The loop is single-instance and can never accumulate:** the 115-tick period is a fixed scoreboard counter (independent of the audio), each `/playsound` self-terminates after 115.4 ticks so even alone the overlap is bounded to the 0.4-tick seam (two copies at most), and `sound_loop` additionally `stopsound`s the old copy an instant before starting the new one ‚Äî so exactly one instance ever plays, and a `/reload` or lag spike mid-sample can't strand a stray copy into a phasing stack. The toggle is state like every mode, but with a config-side default: `.CARTSOUND` (¬ß4.1), seeded once by `modes_init`. Independent of `.HIDECART`. (Bedrock re-creates the same sample in script on the same 115-tick cadence, through its own resource pack's attenuation-free `ir.cart_roll` definition ‚Äî an earlier played-once design that trusted the file's baked-in FMOD loop flag died after a single 5.8 s play in practice, and a copy emitted at a just-joining client was dropped outright; the fixed clock self-heals both within one cycle ‚Äî ¬ß11a/¬ß11e.)

**`function/mode_aggro_on.mcfunction`** / **`mode_aggro_off.mcfunction`**
**Mobs aggro** (`.AGGROMODE`, default on): whether hostile mobs can see the rider. The lever on both editions is an **invisibility effect on the rider** ‚Äî the one vanilla mechanism that reaches mob detection. `_on` (the default) clears it: mobs notice the rider and react like they would to any player ‚Äî creepers sneak up and hiss, skeletons draw their bows, zombies groan and give chase ‚Äî pure ambience, since the rider is untouchable regardless (Resistance 255 + the damage gamerules) and can't be shoved off the rig (a pinned passenger). `_off` applies it (infinite, particle-free): on Java mob sight range shrinks to a few percent (the rail hovers above that radius), on Bedrock invisible players are **completely undetectable** ‚Äî the ride glides through the night unbothered while the scenery mobs still spawn and wander. Side effects of the invisibility: the rider's own body is hidden in third-person/F5 on both editions, and on Bedrock the first-person arm disappears too ‚Äî that arm-hiding used to be its own knob (the retired `.HIDEHAND`, whose invisibility is what made Bedrock mobs ignore every ride, the very bug this mode fixes), so on Bedrock aggro **on** necessarily shows the arm and aggro **off** hides it; the two are physically one setting there. The Java files apply/clear the effect directly on `@a[tag=ir_rider]` (plus `launch_done` at ride start; `stop` clears it with the rider's other effects); on Bedrock the files only flip the score and the keeper asserts the effect once a second (¬ß11e).

**`function/modes.mcfunction`**
Status printout: one `tellraw` line with the rain/sky/hide-cart/sound/mobs-aggro toggles, a second line spelling out the tri-state time mode in words and the adjustable land ride speed (with "(default)" when `.speed` equals `.DEFAULTSPEED`), a third spelling out the tri-state torch mode (off / on day-and-night / auto at night), a fourth naming the torch-density preset, and a fifth naming the Track light state (on/low/off, or "custom (level N)" for a hand-set `.LIGHTMODE`).

**`function/give_menu.mcfunction`** ‚Äî *the pinned hotbar items (menus + speed + tips)*
The per-tick **inventory keeper** (run as each adventure player from `main`): pins the rider's seven hotbar items and polices everything else, so at every tick boundary the items exist and nothing else ever accumulates (`stop` takes them all back). It only touches slots that are actually wrong ‚Äî each pinned slot is re-filled (`item replace`) only while it doesn't hold its own item (matched by `custom_data`; the books carry an `ir_book` marker for this ‚Äî **bump its value whenever a book's pages change**, or mid-ride riders keep the stale copy), and the inventory is wiped (`clear`) only when the occupied-slot count (`.inv`, via `if items entity @s container.*`) exceeds the seven pinned ones (a pickup that slipped past the item sweep, a moved copy; the offhand sits outside `container.*` and is emptied directly). The old unconditional clear + re-give re-fired the client's item-pickup animation every tick, freezing all the icons on its first ‚Äî visibly stretched ‚Äî frame. The layout puts the settings menus on the far left, Debug on the far right, and the speed trio (‚àí, Reset, +) in between, separated by empty slots ‚Äî Speed Reset is the dead center of the hotbar (slot 4):

The books' pages are 1.21.5+-format SNBT text components: clickable links (each a `click_event:{action:"run_command",command:"trigger ir_menu set <n>"}` ‚Äî no leading slash; the page root is an empty `{text:""}` so its style can't inherit into the children) plus short descriptive `hover_event` tips (the concrete `/scoreboard` examples live behind the Debug book's `[Command help]` instead). The Speed items (‚àí, Reset, +) are **re-modeled `carrot_on_a_stick`s** ‚Äî the one item whose `used:` statistic increments on any right-click ‚Äî disguised as a rail / a minecart / a powered rail via the `minecraft:item_model` component and told apart by `custom_data` (`ir_spd` ‚àí1/0/+1 ‚Äî ¬ß6.10); the menu books wear `item_model` disguises the same way (the item stays a `written_book`, so it still opens on use ‚Äî `item_model` is purely visual), and every book carries `enchantment_glint_override=false` ‚Äî a written_book renders with the enchanted shimmer by default, which read as four glowing hotbar icons.

(Bedrock's menus are native `@minecraft/server-ui` forms driven by the script ‚Äî ¬ß11a/¬ß11e.)

**`function/menu_tick.mcfunction`**
The click dispatcher, run from `tick` every tick (ride or no ride, so a click can never go stale): maps the `ir_menu` values ‚Äî 1/2 rain, 3/10/4 time night/day/default, 5/6/30 torches on/off/auto, 7/8 sky, 9 `modes`, 13 speed reset (retired from the book like the old 11/12 ‚àí/+ links ‚Äî the Speed hotbar items cover all three ‚Äî but kept wired for stale book copies), 14/15 debug chat, 16‚Äì20 the sidebar views, 21 command help, 22-25 torch density low/medium/high/max, 26/27 hide cart on/off, 28/29 minecart sound on/off, 31/32 mobs aggro on/off, 33/34/35 track light off/low/on ‚Äî to their functions, executed `as` the triggering player, then `reset`s and re-`enable`s the objective for everyone (a trigger objective disables itself per player after each use, and reset drops the enabled flag with the score ‚Äî so both lines run, in that order). Also fans the Speed items' `ir_click` stat counts out to `speed_click` (¬ß6.10). Both objectives are created by `load`.

*(Bedrock: rain/night/day are the same commands with Bedrock's stable lowercase gamerule names; sky/torches/mobs-aggro only flip the score and `scripts/main.js` does the native work ‚Äî see ¬ß11a.)*

### 6.10 Ride speed & the debug tools

**The adjustable ride speed.** The ride has **three remembered cruise speeds**, one per context, all state like the mode toggles (seeded from their config default by the shared `modes_init` only when unset, then owned by the player): `.speed` for land (default `.DEFAULTSPEED`), `.ocnspd` for the ocean sprint (default `.OCEANSPEED`), `.skyspd` for sky mode (default `.SKYSPEED`). The **Speed ‚àí / Speed Reset / Speed +** hotbar items tune **whichever context is active** ‚Äî sky while `.SKYMODE` is 1, ocean while `.fast` is 1, land otherwise ‚Äî leaving the other two untouched, so entering a context "jumps to its remembered speed" (its config default until you adjust it) and your adjustment is still there next time. One ‚àí/+ click nudges the active speed `.SPEEDSTEP` blocks/s (**4** ‚Äî a fixed cross-edition constant in the shared `consts.mcfunction`, deliberately not a config setting; Bedrock's Ride Settings-form slider sets an exact value instead); the **"Speed Reset"** item (dead center of the hotbar, slot 4, between ‚àí/+) returns the active speed to its config default. The floor is 1, and a +one-step click from that floor rejoins the `.SPEEDSTEP` grid at 4 instead of landing on 5 (so the numbers stay even). Every change prints the new value ‚Äî with "(default)" appended when it equals the active default. `begin` applies `.speed` at ride start, `speed_down` restores it after an ocean sprint, `mode_sky_off` restores it when sky mode ends.

**`consts.mcfunction`** *(shared source: `src/shared/functions/`)*
Fixed cross-edition internal constants ‚Äî numbers the logic needs but nobody tunes per world, kept out of `config.mcfunction` on purpose. Currently just `.SPEEDSTEP` (4). Run at load beside `config` (Java `load`, Bedrock `init()`); Java's fixed-point helpers (`.C16` & co.) stay in `load.mcfunction` because Bedrock does that math in floats.

**`speed_step.mcfunction`** *(shared source: `src/shared/functions/`)*
The speed state machine both editions run ‚Äî and it tunes **whichever cruise speed is active**: the sky cruise `.skyspd` (default `.SKYSPEED`) while `.SKYMODE` is 1, the ocean cruise `.ocnspd` (default `.OCEANSPEED`) while `.fast` is 1, else the land speed `.speed` (default `.DEFAULTSPEED`). It reads `.spdir` (the signed change in blocks/s ‚Äî the items pass ¬±`.SPEEDSTEP`, Bedrock's slider an exact delta, 0 = reset), copies the active target + its config default into the working copy `.spcur`/`.spdef`, runs the math on `.spcur` (reset copies the active default), floors it at 1 (no upper cap ‚Äî vanilla's own gamerule bound is the only Java limit, and Bedrock's pace soft-ceiling keeps an absurd value from outrunning the track), lands a +one-step click from that floor back ON the `.SPEEDSTEP` grid (4, not 5 ‚Äî the `.spfloor` flag; only the exact +`.SPEEDSTEP` request gets the treatment, larger deltas from Bedrock's slider mean an exact value), writes `.spcur` back to the active target, and answers `.spdflt` (1 = the result equals the active default). The APPLY/report is native and reads `.spcur` (so it never re-branches on the context): Java's `speed_apply` pushes `.spcur` into the minecart max-speed gamerule **unconditionally** ‚Äî the active context's speed is by definition the one the gamerule should hold, so a click applies immediately in every context (`speed_up` still re-asserts the ocean cruise each chunk as the safety net); Bedrock's `speed_msg` just prints it (the script reads the three scores as the virtual pace target every tick, so there is nothing to push ‚Äî and it mirrors its script-side sprint flag into the `.fast` score on every transition, `syncFast()`, so this shared file can branch on it).

**`function/speed_inc.mcfunction`** / **`speed_dec.mcfunction`** / **`speed_reset.mcfunction`** / **`speed_apply.mcfunction`**
The three entry points (each sets `.spdir` to +`.SPEEDSTEP` / ‚àí`.SPEEDSTEP` / 0, runs the shared `speed_step`, then `speed_apply`) and the Java apply+report tail. Bedrock has the same three entry-point function files, ending in `speed_msg` (report only) instead.

**`function/speed_click.mcfunction`**

## 7. The algorithms in depth

### 7a. Terrain-surface sampling ‚Üí rolling average
Per column, `sample_window` reads the **terrain** surface Y every `.SAMPLE_BLOCK_INTERVAL` blocks over the next `.SAMPLE_WINDOW` blocks (12 points over 48 at the defaults) and averages them into `.avg` (divisor = the derived count `.winn`). Every read is a two-pass probe (both editions ‚Äî Java's `probe_surface`, Bedrock's `probeSurface()`): first the native highest-block lookup (Java's `motion_blocking_no_leaves` heightmap / Bedrock's `getTopmostBlock` + liquid climb), then a **dig-down through the per-edition not-terrain list** (`#infinite_rail:not_terrain` / `not_terrain.js`) ‚Äî tree trunks, leaves, giant mushrooms, bamboo, and man-made structure blocks (village houses: planks, roofs, glass, wool‚Ä¶) plus the air under them ‚Äî so forests and villages read as the *ground they stand on*, not their canopy/roof line. Water is not in the list: liquid surfaces count as terrain (oceans read as sea level and get bridged). Two safeguards on the samples: void/ungenerated reads (`‚â§ ‚àí63`) are replaced by the previous average, and each sample is **clamped to at most `.DOWNCLAMP` below the previous average** (there is deliberately no upward clamp ‚Äî approaching mountains register at their full height). The down-clamp is the "smoothing" dial for holes: small values make the line ignore sudden dips (they get bridged level); large values make it dive after every hollow.

### 7b. The event model (slope shaping)
The target elevation is `.avg + .HOVER`. Rather than nudging one block at a time, the rail moves in **events**: once it decides to climb or descend, it does so as a single unbroken 45¬∞ run (`.slope` persists; `decide` keeps `.dir` nonzero) until `.railY` reaches the target ‚Äî never "up, flat, up, flat" (a descent may also end early, resting just above ground the next step would cut into ‚Äî ¬ß7j). Between events the rail is flat, and two spacing gaps govern when a new event may start: `.SAMEGAP` (repeat the same direction) and `.TURNGAP` (reverse). `.MIN_CHANGE` adds hysteresis so terrain noise below that height difference is ignored. When a change is *wanted* but a gap forbids it, the rail **holds level** ‚Äî which is exactly what turns into a **bridge** (ground falls away) or a **tunnel** (ground rises into the carve). So bridges and tunnels are not special cases; they emerge from "hold the line until the gap allows a change." On top of this, the **ground-contact guards** (¬ß7j) re-time and re-bound events against the actual surface just ahead, and the **big-event gap credit** (¬ß7k) lets a large event shorten the *next* event's gap ‚Äî the gaps still gate every start, at their (possibly discounted) value.

### 7c. Column geometry (how slopes map to blocks)
`advance` moves the head and picks the place function by `.dir`:
- **Flat:** head east +1; rail at `railY`.
- **Climb:** head east +1; place `ascending_east` at the *current* `railY`; then head up +1 and `.railY += 1`. So each climbing column's rail is one higher than the last ‚Äî a staircase of ascending rails a minecart takes as a smooth 45¬∞ line.
- **Descend:** head east +1 **and down ‚àí1**; place `ascending_west` at the new (lower) `railY`; `.railY ‚àí= 1`. Each column then carves clearance above, lays the redstone support below, sets the rail, and adds the light (¬ß6.6).

### 7d. Power & the disguise
Every rail is `powered=true` and sits directly on a **block of redstone**, which powers it (a rail resting on a redstone power source is activated) with no torch, no support stack, and no barriers. Because a raw redstone block would show red from the side of a bridge, each one is covered by a smooth-stone `block_display` (`ir_disp`). The display needs a `brightness` override (it sits inside an opaque block ‚Üí samples light 0 ‚Üí would be black) and a Y/Z-only oversize (to cover its visible faces without overlapping neighbors). Cost per column: **1 block + 1 display + 1 rail** (down from 5 blocks + 1 rail in the old torch design).

### 7e. Chunk loading / unloading
`forceload` generates a corridor `.TERRAIN_GENAHEAD` blocks ahead of the head so the heightmap scanner always has real terrain, and releases chunks a few hundred blocks behind. There are **two independent look-ahead distances**: `.PACE_CART_BEHIND` (how far ahead of the *cart* the rails are laid) and `.TERRAIN_GENAHEAD` (how far ahead of the *rail head* the world is generated) ‚Äî so terrain exists ‚âà `.PACE_CART_BEHIND + .TERRAIN_GENAHEAD` ahead of the cart. Memory stays flat (passed chunks unload), though vanilla commands can't delete chunks from disk, so the world folder still grows slowly.

### 7f. The keepers
Per-tick guards in `main` make the ride truly unbreakable: anything riding the pace cart that isn't the plug is ejected, as is anything riding the ride cart that isn't a player; a dismounted rider is re-mounted into the ride cart (onto the bare camera seat while `.HIDECART` hides the cart); every mob/creature within 8 blocks of the pace cart is killed (a crowd of animals physically shoves and stalls a minecart; the rider is the rig lead ahead and never sees or hears it ‚Äî Java-only, Bedrock's virtual pace can't collide); water/lava is cleared out of the pace cart's cell and the cell ahead (a wading cart crawls, lava burns); dropped items and XP orbs within 16 blocks of the camera seat are killed before the rider glides into pickup range (the inventory keeper would delete the pickup anyway, but not the pickup *sound*); **hostile projectiles are killed within 6 blocks of the seat** (the `#infinite_rail:projectiles` entity-type tag ‚Äî arrows, tridents, fireballs, potions‚Ä¶; a 0-damage hit on the untouchable rider still plays the player-hurt "oof", and no command can react before the sound's onset, so the hit is prevented instead ‚Äî skeletons still aim and shoot, their arrows just vanish at the last moment; 6 blocks exceeds an arrow's per-tick travel plus the ride's own motion, so nothing crosses the bubble between ticks ‚Äî ¬ß9), and the hurt sound's **tail** is stopsounded every tick for the residual melee/blast case (`stopsound ‚Ä¶ entity.player.hurt` ‚Äî a mitigation, not the fix; deliberately *not* invisibility, which would kill the aggro ambience ‚Äî ¬ß9); the plug and the ride cart are re-mounted onto their perches (unconditional attempts that fail silently while already seated); and if the pace cart's eastward speed ever drops near zero it's re-boosted to `0.5`. The ride cart's pitch is locked horizontally, and the **inventory keeper** (`give_menu`, ¬ß6.9) polices the player's inventory ‚Äî the seven **pinned hotbar items** (the Ride/Visual Settings, Tips and Debug books and the Speed ‚àí/Reset/+ items, ¬ß6.9/¬ß6.10) are re-pinned in place only when missing or wrong, and everything else is wiped the tick it appears, so the items are always there and nothing else ever accumulates. Combined with the always-powered rails, the ride can never stop ‚Äî and because both carts always carry a passenger, neither can be entered by right-click or scoop up passing mobs.

### 7g. The smooth camera (the ride rig)
Java has no `/camera` command (that's Bedrock-only), so the pack uses the vanilla-Java equivalent ‚Äî a riding stack teleported along a smoothed path. The design has three pillars:

1. **One rigid rig, one mount, zero transitions.** The player sits in a real minecart (`ir_ride`) that is itself a permanent passenger of the interpolated camera seat (`ir_seat`). Clients position passengers from their vehicle every frame, so seat ‚Üí ride cart ‚Üí player move as a single rigid body: the cart the player sees can never bounce, tilt or shift against their view, and eye height is genuine minecart-passenger parity by construction ‚Äî no calibration, no mount swaps. The player mounts exactly once per ride; this matters because every player mount event flashes the client's "press ‚áß to dismount" hint, which cannot be suppressed server-side. (Vehicle-swap designs also physically move the player, because passenger attachment offsets differ between entity types ‚Äî the rig sidesteps both problems.)
2. **A constructed S-curve, not a chase.** The pack *built* the track, so it knows the exact elevation profile ‚Äî `advance` records every column's rail Y into a storage list. From it the camera height is **constructed statelessly each tick** as the higher of two candidate curves:
   - `c1`, the S-curve: take `lifted(x) = min(max of the profile over the next ~.CAMLIFT+2 blocks, railY + .CAMLIFT)` ‚Äî the rail line raised by `.CAMLIFT` wherever the track climbs, rising just before climb corners and flattening at the summit level `.CAMLIFT` early ‚Äî then **average it over a symmetric ¬±`.CAMBLEND/2` window**. The average reproduces straight stretches *exactly* (level on flats, truly parallel at 45¬∞ mid-climb ‚Äî no lag, no exponential tail) and turns every corner of `lifted()` into a parabolic blend `.CAMBLEND` long. Result: the camera lifts off ~`.CAMBLEND/2 + .CAMLIFT + 2` blocks before a climb, is already moving parallel when the slope arrives, rides it precisely, then decelerates and lands **level, exactly at the summit height** ‚Äî no 45¬∞-pin, no kink. The blend never stretches across a whole slope, so smoothing can't accumulate into tunnel-roof collisions.
   - `c2`, the descent chaser: the classic reactive ease toward the rail line by `1/.CAMSMOOTH` per tick ‚Äî it floats above the line as the track drops away and settles into valleys (on descents `lifted()` hugs the line, so `c2` wins the max; on climbs it lags below and is ignored). A final floor at the rail line means the rig can never sink into the track.
3. **A hidden cart sets the pace.** The rig rides the rig lead (`.PACE_CART_BEHIND ‚àí .RIDER_BEHIND`) blocks east of the pace cart (`ir_cart`), which rolls along the physical rails behind the viewer, out of forward view. Whatever speed the rails push it ‚Äî including a changed minecart max-speed gamerule under the `minecart_improvements` feature ‚Äî the rig inherits automatically; there is no hard-coded velocity anywhere. The pack sets that gamerule to `.DEFAULTSPEED` at start and to `.OCEANSPEED` over long ocean stretches (¬ß7h), and the rig simply follows.

Because riding only carries *position* (never view), the player keeps full free-look ‚Äî better than Bedrock's `/camera`, which locks the view. The rider is visible, sitting in their gliding cart like on any minecart ride. (The ride cart, being off-rail, doesn't pitch on slopes ‚Äî it glides level through the smoothed climbs, which reads naturally with the eased motion.)

### 7h. The ocean speed-up
A long ocean crossing is the one stretch with nothing to look at, so the ride quietly picks up speed over open water. Each tick `ocean_check` maps the **rider's** X (the seat, the rig lead ahead of the pace cart) to a chunk index (`.rigX / 16`) and acts only when that index changes ‚Äî i.e. once per chunk the rider enters. Sampling at the rider, not the far-behind pace cart, is what makes the speed reflect the water the viewer is actually over. On each new chunk it samples the biome directly under the rider with `execute at ir_seat if biome ~ ~ ~ #minecraft:is_ocean` (the vanilla tag that covers every ocean-named biome) ‚Äî **minus the frozen oceans** (`unless biome` `frozen_ocean`/`deep_frozen_ocean`): pack ice and icebergs are scenery worth watching, so frozen oceans deliberately read as land (Bedrock's explicit `OCEAN_BIOMES` id set simply omits them). Two run counters follow the crossing: `.oceanRun` counts consecutive ocean chunks (any land chunk zeroes it), `.landRun` counts consecutive non-ocean chunks (any ocean chunk zeroes it). Once `.oceanRun` reaches `.OCEANCHUNKS` the ride switches the minecart max-speed gamerule to the **ocean cruise `.ocnspd`** (`speed_up`) ‚Äî adjustable state seeded from `.OCEANSPEED`, tuned by the Speed ‚àí/+/Reset items **while the sprint is on, in both directions** (the shared `speed_step` targets it while `.fast` is 1, and a mid-sprint click applies instantly through `speed_apply`; the old `max(.OCEANSPEED, .speed)` "the ocean never slows the ride" rule is retired ‚Äî each context simply has its own remembered speed now) ‚Äî and keeps re-asserting it every ocean chunk, so the cruise always sticks, even over a manual `/gamerule` change; once back on land, when `.landRun` reaches `.LANDCHUNKS` it drops back to the land cruising speed (`.speed` ‚Äî the config default `.DEFAULTSPEED` unless adjusted with the Speed items, ¬ß6.10) via `speed_down` a single time and then leaves the gamerule alone (so the land speed stays hand-tweakable). The hysteresis (`.LANDCHUNKS` of land before reverting) keeps small islands or gaps from flip-flopping the speed. Because it drives the *same* gamerule the pace cart already obeys, the smooth camera (¬ß7g) inherits the new speed with zero extra work. `.OCEANSPEED 0` disables the whole feature. Like all minecart-speed control, this needs the world's **Minecart Improvements** feature enabled; without it the speed writes are no-ops and the ride cruises at vanilla pace throughout.

### 7i. Vegetation-sparing clearing
The clearance bore no longer flattens everything in its 3√ó(`.TUNNELCLEAR`+1) box. Per cell, the rules are:

- **Critical envelope ‚Äî always cleared:** the rail cell and the cell above it, center only (the cart and rider pass through here), plus everything the column *places* (support below, rail, light at rail+3).
- **Vegetation-sparing ‚Äî everywhere else:** the side cells (Z‚àí1/Z+1, every height) and the center cells ‚â• 2 above the rail are cleared **unless** they hold natural vegetation ‚Äî tree trunks, leaves, giant mushrooms, bamboo, sugar cane, flowers, vines, crops, water plants‚Ä¶ Terrain (stone, dirt, sand) is never spared, so tunnels bore exactly as before; the ride just brushes *through* forests instead of mowing a square canyon.
- **The slope exception ‚Äî full center clear:** the camera floats up to `.CAMLIFT` above the rail line entering, riding and leaving slopes (¬ß7g), so overhead vegetation there would brush the rider. Slope columns, and `.SLOPECLEAR` flat columns on **each side** of every slope, clear their full center bore unconditionally. Vertical only ‚Äî the side cells spare vegetation even there.

The **which-columns logic is shared** (it lives in the same shared `.mcfunction` brain both editions run): `decide` computes `.veg` per column (0 on slope columns, and while `.vclear` ‚Äî armed to `.SLOPECLEAR` by `end_event` ‚Äî counts down after a slope), and `start_event` raises `.retro`, telling the edition's builder to retroactively full-clear the center bore of the `.SLOPECLEAR` columns *before* the slope (they were already built when the slope was decided). `tools/simulate.mjs` asserts the `.veg`/`.retro` contract on both emitted copies.

The **what-is-vegetation list is per edition**: Java's is the `#infinite_rail:keep` block tag (`src/java/data/infinite_rail/tags/block/keep.json`, tested per cell by `carve_layer` with `execute unless block ‚Ä¶ #infinite_rail:keep`); Bedrock's is `src/bedrock/bp/scripts/vegetation.js`, whose `isVegetation()` `placeColumn()` calls on each cell's typeId (Bedrock commands have no block tags). The two files are maintained by hand as a pair ‚Äî the editions' block ids and grouping mechanisms differ anyway (Java has vanilla group tags, Bedrock matches id fragments), so each spells the same policy natively; keep them in sync when adding or removing a plant.

Two deliberate consequences: a tree trunk dead on the centerline keeps its crown (the envelope punches a 2-block gap through it, plus the light cell at rail+3), and spared leaves with no log left in range decay naturally ‚Äî that's vanilla behavior, not a bug.

### 7j. Ground-hugging slope timing (the near scan)

The rolling average (¬ß7a) is good at deciding **where** the line wants to be, but bad at deciding **when** to move: it is a 48-block forward *mean*, so it lags and dilutes around edges. Left alone, that produces three signature uglinesses ‚Äî the line starts a descent while still crossing high ground (trenching down through a mountain's tail to get a head start on the valley beyond), it descends a level or two *into* a valley floor it is about to leave anyway (dip, cruise in a trench, then descend again), and it ends climbs at the crest-diluted average, tunneling right under hilltops.

The fix is a second, much shorter terrain read: the **near scan**. Each edition natively probes the surface every 2 blocks over the next `.SAMPLE_WINDOW` blocks (odd offsets +1, +3, ‚Ä¶; Java: the `near_scan`/`near_step` probe recursion; Bedrock: `nearScan()` over the memoized `surfaceY()` reads ‚Äî effectively free there). The probe itself already digs through trees and structures (the not-terrain pass, ¬ß7a), and consecutive probes are additionally folded into **pairs** ‚Äî `min(this, previous)` ‚Äî to erase what the dig-down can't: a 1-2 block spike of *real* terrain (a rock fin, a lone pillar) only ever catches one probe of a pair, so the min drops it, while real ground (4+ blocks wide) spans both probes and registers. The pairs boil down to three integers handed to the shared brain beside `.target`/`.railY`: **`.gfloor`** (highest pair within `.DOWNLOOK_AHEAD` ‚Äî the ground that governs descents), **`.gmax`** (highest pair anywhere in the walk ‚Äî the climb contact/crest trigger; the climb side has no reach knob, it always scans the full window) and **`.gcone`** (the **climb schedule**: over pairs actually *in the way* ‚Äî above `.railY ‚àí .HOVER`, since ground the line already clears level needs no climb ‚Äî the highest 45¬∞-projection `height ‚àí distance`, i.e. the height the rail must *already* be at for a 45¬∞ ramp from here to crest everything coming). From them the shared `decide`/`consider_start` apply five rules:

- **Descend late** (start veto, `.dig2`): a descent may not *start* without clear runway ‚Äî room for at least two down-steps above the **descent floor**, `.gfloor + .DOWNGRACE`. Wanting to descend while ground within `.DOWNLOOK_AHEAD` is still in the way just holds the level (counting `.flat` like any hold); the descent then begins at the drop-off and glides down in open air. Dips and gaps *narrower* than `.DOWNLOOK_AHEAD` never get descended into at all ‚Äî they are crossed level (bridged), which is also what keeps the line from diving into a slot it would only have to climb straight back out of.
- **Floor stop** (continue guard, `.dig`): a descent in progress **ends** when one more step would land the rail below the descent floor, resting just above the ground it was about to cut into; once the ground falls away, the next descent event carries on ‚Äî ‚â• `.SAMEGAP` later, exactly like any other event. Because every placed descent column was validated against the *tallest* ground it overlooks, **descents physically cannot trench** ‚Äî and because every stop is a real, gap-paced event end, long descents down rough slopes come out as clean 45¬∞ swoops separated by proper benches, never 1-2 column stair-steps.
- **Climb on schedule** (`.due` gate): a wanted climb is *held* ‚Äî even with the average begging for one ‚Äî until the rail is within `.UPEARLY` blocks of the height the 45¬∞ cone demands (`.gcone + .HOVER`). This is what stops the line ramping up 30+ blocks before a mountain just because the 48-block average saw it coming: ramps start `height + ~.UPEARLY` before the crest and top out `~.UPEARLY` columns early. The held columns keep counting `.flat`, so waiting can never cause a gap-block later; and the multi-feature case resolves itself ‚Äî the *nearest binding* crest dominates the projection until a taller, farther peak's projection overtakes it.
- **Climb early** (deadband override): when the level line would physically plow into terrain the scan sees (`.gmax > .railY`), the average agrees the ground is rising (`.diff ‚â• 1`) and the schedule agrees (`.due`), the climb is wanted even though `.diff` is still inside `.MIN_CHANGE`. The spacing gaps still have the final say.
- **Crest completion** (`.push`): a climb in progress keeps climbing until it rides a full `.HOVER` above the highest ground the scan sees (`.railY < .gmax + .HOVER`), up to `.UPGRACE` blocks above the target. Wide hilltops (whose beyond-crest downslope dilutes the average below the summit) get ridden *over* at proper hover height instead of tunneled just under ‚Äî without this the climb would stop a block low and park there inside the deadband; anything taller than the `.UPGRACE` budget still gets punched, so narrow rock fins don't turn into bobbing.

Priorities and safety: the gaps always win (`.SAMEGAP`/`.TURNGAP` gate every event exactly as before ‚Äî the guards only hold events back, stop them early, or extend one already running; they never place events closer together than the gaps allow; the two things that *lower* a gap itself are the big-event credit, ¬ß7k, and the stretch shift, ¬ß7l ‚Äî both part of the gap computation, not bypasses); sky mode bypasses all five rules (it holds `.SKYY` dead level and punches through, as documented); and the sentinels fail safe ‚Äî `.gfloor`/`.gmax` arrive as `‚àí10000` on no data (their guards pass, plain event behavior), `.gcone` as `‚àí10000` when nothing ahead needs climbing (the gate holds ‚Äî nothing to be due for) and `+32000` when the scan had no data at all (the gate never holds). Setting `.DOWNLOOK_AHEAD` to 0 disables the descent side wholesale (the climb side is always on ‚Äî its sentinels fail open on no data); `.UPGRACE 0` disables only the overshoot; `.UPEARLY ‚â• ~50` disables only the schedule.

`tools/simulate.mjs` locks the behavior in: it feeds the same near-scan values to both editions' emitted brains, asserts no descent column ever lands below the descent floor, that descents never start without two-step runway and never end while the floor below is clear, that climbs never start ahead of schedule, allows climb starts inside the deadband only with logged ground contact, and runs three purpose-built terrains ‚Äî `mesa` (the line must cross a high tabletop level and descend only at the drop-off, where it used to trench down through the last ~45 columns of the top), `ridge` (a narrow ridge diluted to `.diff = 1`, reachable only through the early-climb + crest-push path) and `hillside` (a long 1:2 downhill face descents must take as gap-paced 45¬∞ swoops without ever entering the ground). (Two more purpose-built terrains exercise the gap credit ‚Äî ¬ß7k.)

### 7k. The big-event gap credit

Long spacing gaps (the 50/50 defaults) are what make ordinary terrain read calm and stately: small dips get bridged, small bumps get tunneled, the line never bobs. But applied blindly they misfire around **large** elevation changes, in two signature ways: reaching the top of a tall narrow mountain, the line must run a full `.TURNGAP` of flat clifftop bridge past the peak before it may reverse and come back down; and one big ascent too tall for a single event (the average keeps rising after each climb tops out) gets its chained climbs spread `.SAMEGAP` apart, tunneling into the mountainside between them.

The fix is a credit: **a large event earns the *next* event an earlier start.** `decide` counts every sloped column into `.evrun` (`start_event` restarts it, so during a flat stretch it holds the finished event's size ‚Äî at 45¬∞ the column count *is* the height). When `consider_start` computes the required gap, it subtracts `.evrun √ó .GAPRATIO / 100` (the knob is a **percent** of the event's height ‚Äî the √ó100 fixed point, via the shared `.C100`, is how an int-only scoreboard carries fractional ratios; default 50: a 40-block climb shortens the next gap by 20 columns; 67 ‚âà a 1/1.5 ratio; 0 = the credit is off), floored at 0. One guard keeps it honest ‚Äî the **worth-it test**: the newly wanted move must itself be at least `.evrun √ó .GAPMATCH / 100` blocks (also a percent; default 50: at least half the big event's size; 0 = no requirement), or the full gap applies ‚Äî so a big climb lets the line *descend off the mountain* early, never lets a 3-block bob sneak in at the summit. The move's size witness is `|.diff|`, except for climbs, where the rolling average structurally *dilutes* an approaching rise to roughly half (the schedule starts climbs while the 48-block window still spans the low ground in front); climbs therefore also consult the near scan ‚Äî `.gmax + .HOVER ‚àí .railY`, the climbing actually needed ‚Äî and use the larger of the two.

Properties that fall out of the arithmetic, all locked in by the simulator: small everyday events earn next to nothing (`.evrun` of 2-4 ‚Üí a credit of 1-2 columns), so typical terrain keeps the full gaps ‚Äî `rolling`'s decisions shift by at most a column; an absurdly large credit (a 100-column sky-mode glide) demands an equally large follow-up or none at all, so it can't zero the gaps for ordinary wiggles; and a credit can never go stale in a harmful way ‚Äî once enough columns pass, `.flat` exceeds the *full* gap anyway and the discount changes nothing, so it only ever acts close behind the event that earned it, which is exactly the "recent big feature" semantics wanted. The ground-contact guards (¬ß7j) still apply unchanged on top: a discounted descent still needs two-step runway, a discounted climb still waits for its schedule. Two purpose-built simulator terrains assert the credit actually engages: `peak` (a 40-tall 1:1 triangle mountain ‚Äî a reversal must start inside the full `.TURNGAP`, descending off the summit ~16 columns sooner) and `bigsteps` (two 35-block walls 80 apart ‚Äî the second, same-direction climb must start inside the full `.SAMEGAP`; it also stops the second climb starting gap-late and cutting into the wall).

### 7l. The stretch shift (descents)

Even with the credit (¬ß7k), a gap-blocked descent still waits out a shortened-but-real gap up on its clifftop bridge. The stretch shift is the "logical second pass" over the plan: when a descent is wanted but gap-blocked, the native **shift scan** (Java `shift_scan`/`shift_step`, Bedrock `shiftScan()` ‚Äî run per column between the near scan and `decide`, cheap because it only probes while the question is live) verifies the *entire* planned move against the terrain ahead, before anything is built:

1. **The whole shifted 45¬∞ path stays clear of ground** (by `.DOWNGRACE`, pair-mins like the near scan): the floor guard can then never fire mid-descent, so the shifted descent is provably the **same single event** ‚Äî no new plateaus, no split into extra descents.
2. **The landing really is a stretch**: `.GAPSTRETCH` columns of ground sitting *at* the landing level (within `.MIN_CHANGE` below the hover line). Ground still falling away past the landing fails ‚Äî a gentle downhill face is not a landing, and keeps its gap-paced swoops (the simulator's `hillside` mid-slope waits are unchanged; only its final descent onto the plain shifts). This band also forces the average to have converged onto the true bottom first, which is what guarantees the shifted descent lands **at the same level** the un-shifted one would have.

One integer crosses into the shared brain: **`.sver`**, the verified horizon in blocks (written every column; 0 = no). `consider_start` jumps the gap (`.need = 0`) when `.sver ‚â• |.diff| + .GAPSTRETCH` ‚Äî with the same **worth-it** philosophy as the credit: the descent must be at least `.need √ó .GAPMATCH / 100` tall, so a verified but tiny dip can't jump a big gap and busy up gently rolling terrain (the simulator's `rolling` decisions are identical with the shift on and off). Climbs deliberately have no shift: a climb that *starts* as late as possible is the schedule's job (¬ß7j, `.due`), and a gap-late climb is what the credit is for. The rationale mirrors the gaps' purpose: the calm the gap exists to guarantee simply happens at the bottom of the drop instead of on a bridge overshooting the peak. `.GAPSTRETCH 0` disables the shift; the scan reaches at most 96 blocks, so descents deeper than `96 ‚àí .GAPSTRETCH` never shift (they wait out their gap as before ‚Äî fail-closed, like every missing/unset `.sver`). With `.GAPSTRETCH ‚â• 48` (the default 50 qualifies) the verified horizon also covers every per-column floor-guard window along the whole descent, which is what makes the no-split guarantee airtight.

`tools/simulate.mjs` mirrors the scan and the rule exactly, and `peak` asserts the shift engages (a descent jumps the remaining gap onto the verified stretch) **and** that the whole ride stays exactly one climb + one unbroken descent.

## 8. Tuning

All knobs live in `config.mcfunction` (see the group lists and table in ¬ß4.1). **To apply edits: change the value, then run `/reload`** (or rejoin the world) ‚Äî the game re-reads the file and re-runs `config`, updating a ride already in progress. To experiment without editing the file, set a score live **in the knob's `cfg_*` objective**, e.g. `/scoreboard players set .HOVER cfg_terrain 8` (takes effect on the next column; wiped on the next `/reload`/rejoin) ‚Äî the Debug book can put any whole group on the scoreboard sidebar while you tune (¬ß6.10). ## 9. Limitations & gotchas

- **Disk usage grows.** Commands can unload chunks (memory stays flat) but can't delete them from disk, so a very long ride slowly grows the world folder.
- **Single rider.** One cart, one occupant; designed for a solo viewer.
- **Overworld only.** The Nether's bedrock ceiling confuses surface heightmaps.
- **Very low `.HOVER`.** The redstone support is immune to water, but the *rail* is not ‚Äî at `.HOVER 0` or below, the rail itself can sit in water and wash out. Keep the track hovering above sea level. (The power source is safe regardless.)
- **Pack-ice tunnels.** The default `light[level=11]` block is exactly at the ice-melt threshold, so it doesn't melt ice; the redstone block emits no light. So the power stays safe ‚Äî and every Track light preset (11/8/0, ¬ß6.9) is ice-safe ‚Äî but a hand-set `.LIGHTMODE` above 11 could melt ice into the bore.
- **Display entities accumulate** in the built (and saved) chunks like any block; they unload behind the ride with their chunks. `brightness:{sky:15,block:15}` is full-bright, so the disguised stone won't dim at night ‚Äî lower `block` toward 0 in `support.mcfunction` if that reads as too bright.
- **Track history grows.** The camera's profile list gains one int (~4 bytes) per column for the life of a ride ‚Äî a few MB after a multi-day ride. It's reset on every fresh `start`.
- **Sub-block camera math degrades past X ‚âà ¬±2,147,000.** The cart's X√ó1000 fraction read overflows a scoreboard int out there (~3 days of continuous riding); the camera would get a garbage sub-block fraction (‚â§1 block of jitter, track itself unaffected). Everything else uses NBT doubles.
- **Updating the pack over a ride in progress** leaves the camera idle (no track history exists for the already-built line). Run `start` again to begin a ride with the full system.
- **The pace cart is visible looking backward** ‚Äî an empty-looking minecart rolling the rig lead (`.PACE_CART_BEHIND ‚àí .RIDER_BEHIND`, 64 at defaults) behind the viewer. Lower `.RIDER_BEHIND` (or raise `.PACE_CART_BEHIND`) to push it further out of sight ‚Äî keep the lead at least ~40, and `.PACE_CART_BEHIND` below ~250 so the rolling forceload never releases the pace cart's chunk.
- **The vanilla dismount hint** ("press ‚áß/left-ctrl to dismount") is a client-side toast shown on every player mount event; it cannot be hidden by a server or data pack. The rig design means it appears exactly once, at ride start (and again only if the rider dismounts themselves and is re-caught by the keeper).
- **Auto-start on upgraded worlds.** `.autodone` didn't exist before the smooth-camera update, so a pre-existing world that had used the pack will auto-start once on its first load after upgrading (its `.autodone` is unset). Run `stop` once, or set `.AUTOSTART 0`, if that's unwanted.
- **The pack raises the command-chain budget gamerules.** `load` sets `maxCommandChainLength` and `maxCommandForkCount` to 1,000,000 (defaults 65,536) as headroom for heavy config values ‚Äî the launch itself no longer depends on it (it is phased across ticks, ¬ß6.3), but a silently-truncated chain is undebuggable, so the budgets are kept far away. Like the `setup_world` rules, this persists in the world until changed by hand.
- **`load` also turns `sendCommandFeedback` off** (via the same version-picked `names`/`set_rule` plumbing ‚Äî `send_command_feedback` on 26.x). `setup_world` already did this at ride start, but the menu books' click links run `/trigger` **as the player**, and with the rule on every click echoed vanilla's own feedback ("Triggered [ir_menu] (set value to N)") into chat on top of the pack's `[Scenic Rail]` reply. Setting it on every load closes the gap for worlds where the rule was still (or turned back) on.
- **File edits need `/reload`.** See ¬ß8 ‚Äî the single most common point of confusion.
- **The minecart sound is a re-triggered sample, not the engine's loop.** Neither ride cart moves on rails, so the engine's own velocity-driven sound never plays (¬ß6.9); both editions instead re-play vanilla's first-person `minecart/inside` sample at the rider on a clock matched to its length. Consequences shared by both: the loop is a **fixed cadence** ‚Äî it doesn't pitch up on ocean sprints or fade at low speed like a real cart's ‚Äî and there is a faint restart seam every ~5.8 s on both editions. On **Java** it is a pure `/playsound` at a large volume (so it never fades as the ride moves ‚Äî no resource pack, nothing to install); on **Bedrock** it is the RP's attenuation-free `ir.cart_roll` definition, re-triggered on the same clock. Toggle either from the Ride Settings menu (`Sound: [On]/[Off]`), or change the default with `.CARTSOUND`.
- **Ride modes persist on purpose.** The `mode_*` toggles (¬ß6.9) survive `stop`, `/reload` and rejoins, and rain/night mode set plain vanilla world state (weather-/daylight-cycle gamerules + `/weather`/`/time`) that nothing in the pack unwinds automatically. Run the `_off` functions to restore vanilla behavior; uninstalling the pack while rain/night are on leaves the cycles frozen until re-enabled by hand.
- **Minecart speed & the feature flag.** `.DEFAULTSPEED` and the ocean speed-up (¬ß7h) drive the minecart max-speed gamerule, which exists only with the **Minecart Improvements** feature. The pack **enables that feature itself** (`features.enabled` in `pack.mcmeta`), so the gamerule is present whenever the pack is loaded ‚Äî no manual experiment toggle needed. The rule is named `minecartMaxSpeed` on formats 82-91 and `max_minecart_speed` on 92+ (renamed in 25w44a); `names.mcfunction` (base vs `overlay_snake`) supplies the right name into `rule` and `set_speed` runs only that one (a macro line that expands to an unknown gamerule would abort the function, so the wrong name is never emitted). If a speed change still doesn't take, set `.DEBUGMODE 1` ‚Äî it prints the speed being set and the pace cart's real `Motion[0]√ó100` each chunk.
- **Recipe toasts & tutorial hints.** Neither edition ever shows a "recipes unlocked" toast mid-ride: Bedrock turns unlocking off outright (`recipesunlock` / `showrecipemessages` false in its `setup_world`); Java has no such gamerule, so `begin` pre-unlocks every recipe (`recipe give @s *` ‚Äî one combined toast at ride start, then nothing left to unlock). Tutorial-hint toasts are disabled on Bedrock (`gametips disable`, also `setup_world`); on **Java they cannot be touched** ‚Äî the tutorial lives client-side in `options.txt` (`tutorialStep`), no command reaches it. In practice it only affects brand-new installations, and completes itself permanently once done.
- **Torch mode on ice.** Torches now plant on frozen ground too (that's the point), but a torch's light can melt regular ice under/next to it on a later random tick ‚Äî the torch pops (silently, no drop: `doTileDrops` off) and leaves a water hole in the frozen surface. This happens behind or beside the ride and reads as natural thaw; the alternative was frozen biomes staying entirely unlit.
- **The rider takes no damage, but the hurt sound needs suppressing (without invisibility).** The rider is functionally invulnerable ‚Äî Resistance 255 + the `fallDamage`/`fireDamage`/`freezeDamage`/`drowningDamage` gamerules + `mobGriefing false` mean every hit lands as **0 damage**. But in aggro mode (the default) mobs still notice and swing at / shoot the untouchable rider ‚Äî that ambience is the point ‚Äî and a **fully-absorbed 0-damage hit still plays the player-hurt "oof"**. There is no way to *cancel* a damage event from a data pack or the stable Script API, and the editions differ in what they *can* do: **Bedrock** silences `game.player.hurt` outright in its resource pack (¬ß11e) ‚Äî clean and complete. **Java** has no resource pack, and a `stopsound`-only approach **failed in practice**: a function can only react at the next tick boundary, by which time the sound's onset ‚Äî the part you actually hear ‚Äî has already played (cutting the tail is imperceptible). So Java **prevents the hits themselves** where it can: `main` kills every hostile projectile within 6 blocks of the seat each tick (the `#infinite_rail:projectiles` tag, `required:false` ids so entity renames degrade gracefully) ‚Äî arrows/tridents/fireballs/potions are the dominant damage source at hover height, and skeletons still draw and shoot; their arrows just vanish silently at the last moment. The per-tick `stopsound` stays only as a tail-cutter for the residual cases a sweep can't prevent (a melee reach-up on high ground, a creeper that completes its fuse ‚Äî rare at ride speed, since fuses abort as the ride pulls away). Neither lever is invisibility, which would blind mobs and kill the aggro ambience. (Fire/freeze/drowning hurt-sound variants never fire ‚Äî their damage types are gamerule-disabled.)
- **The rider's held item is hidden by inventory clearing; the bare arm follows the Mobs-aggro mode on Bedrock.** Both editions keep the rider's inventory empty every tick, so nothing is ever held. On **Bedrock** the bare first-person arm can only be hidden by an invisibility effect (`/hud` has no `hand` element) ‚Äî and that same effect is what decides whether mobs can see the rider at all, so the arm is now tied to the **Mobs aggro** mode (¬ß6.9): aggro off = invisible = arm (and F5 body) hidden; aggro on (the default) = visible = arm shows. (The old standalone `.HIDEHAND` knob is retired for exactly this reason ‚Äî "hidden arm" and "mobs can aggro" were never physically separable.) On **Java** invisibility doesn't hide the first-person arm, so the Java rider keeps their empty arm either way.

## 11. The Bedrock Edition port & the shared codebase

The repository is a monorepo: `src/shared/functions/` + `src/java/` build the Java data pack documented above, and `src/shared/functions/` + `src/bedrock/` build a native **Bedrock behavior pack** (`tools/build.mjs`; see `BUILDING.md` for the workflow). The port is not a transliteration of the Java files ‚Äî it is the same *design* re-implemented on Bedrock's strengths, sharing the one part that is pure algorithm.

### 11a. The logic boundary: what is shared and what is native

**Shared (identical `.mcfunction` source, both editions):** the event-model brain ‚Äî `decide`, `consider_start`, `start_event`, `end_event` ‚Äî plus `config`, `modes_init` (the ride-mode toggle + `.speed`/`.ocnspd`/`.skyspd`/`.torchdens`/`.LIGHTMODE` seeding, ¬ß6.9/¬ß6.10), `consts` (fixed cross-edition constants like `.SPEEDSTEP`, ¬ß6.10), `speed_step` (the adjustable land/ocean/sky ride speed's state machine, ¬ß6.10), `torch_auto` (torch mode's is-it-night placement gate, ¬ß6.9) and `debug_state` (the Live state sidebar mirror, ¬ß6.10). These are pure scoreboard math ‚Äî runtime state in the `ir` objective, the tunables in the three `cfg_*` objectives (¬ß4.1). Each engine boils its world down to six integers per column (`.target`, `.railY`, the near-ground scan's `.gfloor`/`.gmax`/`.gcone` ‚Äî ¬ß7j ‚Äî and the stretch-shift scan's `.sver` ‚Äî ¬ß7l), calls `decide`, and reads back one integer (`.dir`). All event state (`.slope`, `.flat`, `.lastDir`, the gap rules, the deadband, the ground-contact guards) lives *only* inside the shared files, so the slope-shaping behavior of the two editions cannot drift apart. `tools/simulate.mjs` enforces this in CI by interpreting both emitted copies over synthetic terrains and failing if their decisions ever differ.

The carve-mode logic rides along in the same shared files: `decide` computes `.veg` (may this column spare vegetation? ‚Äî ¬ß7i), `end_event` arms the `.vclear` after-slope buffer, and `start_event` raises `.retro` (the before-slope retro-clear request), so the two editions cannot disagree about *which* columns clear what. The list of *what counts as vegetation*, by contrast, is **per edition by design**: Java's `#infinite_rail:keep` block tag (`src/java/.../tags/block/keep.json`) and Bedrock's `scripts/vegetation.js` module are independent hand-maintained files ‚Äî Java tests cells against the tag in commands, Bedrock calls `isVegetation()` in script, and since the editions' block ids and grouping mechanisms differ anyway (vanilla group tags vs typeId fragment matching), each file spells the shared *policy* in its own edition's terms. A second list pair follows the same pattern: the surface probe's **not-terrain dig-down list** (¬ß7a) ‚Äî Java's `#infinite_rail:not_terrain` tag (`tags/block/not_terrain.json`, which nests `#infinite_rail:keep` and adds air + the man-made structure blocks) and Bedrock's `scripts/not_terrain.js` (which delegates to `isVegetation()` and adds the man-made fragments). Keep both pairs in sync when changing what the carve spares or what counts as terrain; the build fails if any of the four files is missing from its pack.

The shared files are **byte-identical on both engines** ‚Äî the build injects them verbatim (and lints them against the dual-dialect subset), so they can be symlinked from `src/shared/functions/` straight into a dev world. Two conventions make that possible: score holders use the `.` prefix in *both* editions (`#` is a Java-only fake-player convention; `.` is the prefix documented to parse on Bedrock, and Java accepts it just as well), and shared-to-shared function calls use bare **`ir_*` bridge names** instead of an engine-specific path (Java `infinite_rail:name` vs Bedrock `infinite_rail/name`). A bare `function ir_end_event` resolves on Java in the `minecraft` namespace and on Bedrock from the `functions/` root; each edition keeps a one-line trampoline there (`src/java/data/minecraft/function/ir_*.mcfunction`, `src/bedrock/bp/functions/ir_*.mcfunction`) that hops into the real shared file. Three calls are bridged: `ir_consider_start`, `ir_start_event`, `ir_end_event`. A live tweak is the *same command* on both editions: `/scoreboard players set .HOVER cfg_terrain 8`.

**Native per edition (same job, different machinery):** everything that touches the engine. Java's implementations are described in ¬ß6‚Äì¬ß7; Bedrock's counterparts all live in `src/bedrock/scripts/main.js` (stable `@minecraft/server` Script API ‚Äî no experiments, no betas):

| Job | Java mechanism (kept) | Bedrock mechanism (replaces it) |
| --- | --- | --- |
| Terrain-surface sampling | `ir_probe` marker + `probe_surface` (`execute positioned over motion_blocking_no_leaves`, then the `probe_down` recursion digging through `#infinite_rail:not_terrain` ‚Äî trees, structures, the air under them ‚Äî until real ground/liquid) | `dimension.getTopmostBlock()` + a walk down past leaves/foliage *and the `not_terrain.js` list* (same dig-down policy) + a climb back up any liquid column ‚Äî Bedrock's topmost-block probe **skips liquids**, so an ocean read lands on the sea *floor*; the climb restores Java's liquids-count-as-surface semantics, so oceans read as sea level and get bridged instead of dived into. Reads are memoized per column (the sliding window re-samples each X twelve times) |
| Near-ground scan (slope timing, ¬ß7j) | `near_scan`/`near_step` ‚Äî a probe recursion at odd offsets +1, +3, ‚Ä¶ pairing consecutive reads (min ‚Äî residual 1-2-wide real-terrain spikes vanish) and folding them into `.gfloor`/`.gmax`/`.gcone` with scoreboard max operations | `nearScan()` ‚Äî the same odd-offset pair loop over the memoized `surfaceY()` reads (the 48-block window already fills the memo, so the scan costs no extra real probes) |
| Track history | storage `infinite_rail:track y` list + `cam_get` macro (NBT paths need literal indices) | a plain JS array (`trackY`), trimmed behind the ride and persisted (below) |

### 11b. The Bedrock rig and camera

The rig is three pieces like Java's, but with **exactly one mount in the whole system**: the rider sits on the invisible **camera seat** (seat offset 0.35 up), and the **minecart-look cart prop** (tag `ir_ride`) is *not mounted on anything* ‚Äî `cam_move` glides seat and cart **independently, in lockstep**, with the same velocity drive toward the same target each tick. Java stacks seat ‚Üí cart ‚Üí player instead; Bedrock cannot, for two hard-won reasons: mount *state* is not reliably queryable there (the `minecraft:riding` component and `rideable.getRiders()` both under-report, which turned a "re-mount if unseated" keeper into a per-tick mount war ‚Äî pose flicker, mount-sound spam), and the engine proved unwilling to keep an *entity* passenger seated at all ‚Äî the cart kept being ejected within ticks of a successful `addRider` and parked at the dismount spot above the rider's head. Player-on-seat is the one mount Bedrock keeps stable, so it is the only one used. The seat is a tiny custom entity (`infinite_rail:seat`: no gravity, no collision, one player seat). The **cart is a custom entity too** (`infinite_rail:cart`): its client definition uses a re-based copy of the vanilla minecart geometry (`geometry.ir_cart` ‚Äî every cube shifted 16px down, because the vanilla model draws a block high outside the engine's internal minecart renderer) with the vanilla minecart texture, so it looks like a real cart, but it carries none of the minecart's client-side behavior ‚Äî which matters because Bedrock clients tilt a *real* minecart's model 45¬∞ whenever it occupies a block cell containing an ascending rail, even off-rail; the rig glides right along the track line, so at slope entries/exits a real ride cart visibly flickered between tilted and flat. The prop has **no `rideable` component and no `health`** ‚Äî nothing can ever enter it, and Bedrock's mount-health HUD (which showed as rows of hearts over the food bar for a 100-HP vehicle) never appears. A vanilla minecart remains the spawn fallback for an outdated BP. The cart being the seat's *passenger* is load-bearing ‚Äî passengers run no physics of their own, so the engine's minecart logic (capture onto the powered rail in the cart's own block cell, gravity, ground contact) can never fight the script for control of the cart; that fight is exactly what made a directly-driven cart visibly bob up and down. The script computes the same smoothed height `sy` as Java (¬ß7g, float port) and glides the seat toward `(paceX + the rig lead, sy + 0.062 + .CAMHEIGHT/10, centerZ + 0.5)` by setting its velocity each tick; the client renders that as smooth motion, and the player's normal first-person camera rides along ‚Äî **full native free-look with zero added latency**, the same experience as Java.

Why not the `/camera` (Camera API) rig by default? Bedrock's `minecraft:free` preset **does not follow look input** ‚Äî the official camera-system docs state input keeps rotating the *player*, not the detached camera. A Camera-API rig therefore needs the script to pass `player.getRotation()` back into `setCamera` every tick, which adds a perceptible beat of look latency. That trade is available as **`.CAMMODE 1`** (cinematic mode): the view detaches onto `minecraft:free` at eye height above the cart, eased ~0.15 s Linear per tick for extra positional glide, rotation passed through from the player. `.CAMMODE 0` (default) keeps the native camera.

Keepers (the Bedrock subset of ¬ß7f): strangers are ejected from the seat; a dismounted rider (survival mode, or adventure from a pre-survival save) is re-seated; the rig is re-summoned if it ever goes missing; the rider's inventory is kept empty apart from the pinned items (re-pinned only when a slot is missing/wrong ‚Äî never blanket-rewritten, so no pickup-animation flicker); dropped items and XP orbs within 16 blocks of the seat are removed before the rider reaches pickup range (`sweepDrops()` ‚Äî the pickup *sound* was the problem, the pickup itself was already deleted); the cart's cell and the one ahead are kept free of water/lava (`clearCartLiquids()` in `camMove`); a lost cart prop is rebuilt cart-only at the rig (a ~1 s anti-duplicate grace; never via the whole-rig respawn, which would unseat the rider ‚Äî that path now fires only for a missing seat); and once a second the rider's invisibility effect is asserted to match the **Mobs aggro** mode (¬ß6.9): re-applied while `.AGGROMODE` is 0 (mobs blind to the rider; also hides the first-person arm and F5 body), removed while it is 1 (mobs react naturally; the arm shows).
**The rider re-mount decision is positional, never API-queried**: a seated player is pinned to the seat while the rig glides east at cruising speed, so a genuine dismount shows up as distance from the seat that keeps growing tick after tick ‚Äî only a sustained streak (`ASTRAY_TICKS`) triggers a re-mount. The riding component and the rider list both under-report on Bedrock, and re-mounting an already-seated passenger re-fires the mount (the pose-flicker war described above), so neither is ever trusted for mount state. The cart prop needs no keeper at all ‚Äî `cam_move` owns its motion. The plug, stall re-boost, and pace-cart ejections have no Bedrock equivalent because the virtual pace made them obsolete.

### 11c. Speed without the gamerule

Bedrock has no minecart max-speed gamerule, so `.speed` (the adjustable ride speed, default `.DEFAULTSPEED` ‚Äî ¬ß6.10) and `.OCEANSPEED` steer the **virtual pace speed** directly: `ocean_check`'s shared trigger logic (same per-chunk cadence, same `.OCEANCHUNKS`/`.LANDCHUNKS` hysteresis, sampled at the rider) sets a target speed in blocks/tick, and the pace gains or sheds 0.4 blocks/s of speed per tick (`ACCEL` ‚Äî the default 8 ‚Üí 32 ocean ramp takes ~3 s) ‚Äî reproducing the gradual physics acceleration the Java cart gets from its rails. **User speed changes ease faster than context switches**: `tickPace` watches for the target moving while the context (land/ocean/sky) did *not* ‚Äî that is a Speed ‚àí/Reset/+ click or the settings slider ‚Äî and takes a brisk 2.5 blocks/s-per-tick ramp (`ACCEL_CLICK`, a 32 ‚Üí 8 reset lands in ~0.5 s) until the new target is reached, because Java applies a click straight to the gamerule and its deceleration is near-instant; the ocean/sky context transitions keep the gentle rail-physics ramp, where gradual is the point. Consequently the land speed is *continuously* honored on Bedrock (nudge `.speed` with the Speed items ‚Äî or `.DEFAULTSPEED` in config ‚Äî and the ride adjusts within a second), whereas Java applies it per change via the gamerule.

### 11d. State & persistence

The shared brain's state (`.slope`, `.flat`, `.lastDir`, all config) lives in the scoreboard, which Bedrock persists in the world save exactly like Java. The script's own state (headX, railY, centerZ, avg, the pace position and speed, the ocean counters, the descent chaser, the rider's name, and the last 1024 columns of track history) is saved to a world **dynamic property** (`ir:state`, a few KB of JSON) every 2 seconds and on every lifecycle change ‚Äî so a Bedrock ride **survives quitting and rejoining the world**, resuming where it left off. `.autodone` lives there too, so auto-start stays once-per-world across rejoins. The in-memory history is trimmed to the last ~2048 columns (the camera only reads a few hundred around the rig), so an endless ride can't grow memory forever ‚Äî unlike Java's storage list (¬ß9), which is unbounded by design.

### 11e. Bedrock-specific behavior differences & gotchas

- **The support is a custom block, not a disguised redstone block.** Bedrock has no `block_display` entities, so instead of Java's disguise-over-power two-parter the port defines `infinite_rail:support` (BP `blocks/support.json`): a full cube rendered with the **vanilla smooth-stone texture** (the RP's `terrain_texture.json` maps a shortname onto vanilla's `textures/blocks/stone_slab_top` ‚Äî no texture is shipped) that carries **`minecraft:redstone_producer`** at power 15 (`strongly_powered_face: up`), so it powers the rail exactly like a block of redstone. Water immunity and zero light emission are the same as the redstone block it replaces; the script falls back to a bare `minecraft:redstone_block` if the custom block fails to resolve (outdated BP). Track built by older pack versions keeps its redstone blocks.
- **The powered rails are disguised as regular rails** (Bedrock-only cosmetic). The same no-texture-shipped trick as the support: `terrain_texture.json` entries merge per-key across packs, so the RP overrides vanilla's **powered-state** golden-rail shortname (`rail_golden_powered`) onto vanilla's `textures/blocks/rail_normal`, and the track reads as a plain old railway instead of a gold-studded one ‚Äî every track rail renders that state, because the `infinite_rail:support` block below strongly powers the rail above it (and the builder places them `rail_data_bit: true` besides). The **unpowered** shortname (`rail_golden`) is deliberately left vanilla: it is also the golden-rail **item icon**, and remapping it turned the "Speed +" hotbar item into a plain rail. The *straight* regular-rail texture is always the right one ‚Äî this track never turns, and flat and ascending rails share the straight texture. Two consequences: the disguise is **global** (any *powered-state* golden rail anywhere looks like a regular rail while the RP is active ‚Äî including hand-built ones after `stop`; an unpowered one shows its vanilla gold), and it is **visual only** (the block is still `golden_rail`; physics, redstone and pickaxe drops are untouched). Java keeps its honest powered-rail look ‚Äî a Java *data* pack can't retexture anything, and shipping a Java resource pack is out of scope by design (¬ß1).
- **The RP silences four vanilla sound events and defines one of its own.** `sounds/sound_definitions.json` overrides `step.stone`, `step.grass`, `minecart.base` **and `game.player.hurt`** with volume-0 entries (sound definitions merge per-key across packs, whole-entry replace ‚Äî the same mechanism as the texture disguises) to kill the phantom footstep/cart noises the gliding rig produced and the player-hurt "oof" (see the damage-sound note in ¬ß9 ‚Äî the rider takes no real damage through Resistance 255 + the damage gamerules, but a fully-absorbed hit still played the sound; silencing it is Bedrock's clean answer, where Java has to `stopsound` it per tick). `game.player.hurt` is the generic hurt sound (files `sounds/damage/hit1-3`); the fire/drown/freeze variants have their own events but those damage types are gamerule-disabled, so they never fire. The overrides are **global** while the RP is active (any player's stone/grass footsteps, any minecart's rolling loop, any player's hurt sound ‚Äî acceptable for a single-rider pack, the same tradeoff as the other silences). The pack's own ride sound (`.SOUNDMODE`, ¬ß6.9) therefore lives under a pack-private id: **`ir.cart_roll`** points at vanilla's real `sounds/minecart/inside` file (no audio shipped) with **`min_distance` 512** ‚Äî no distance attenuation within that range, so a playing copy holds constant volume as the ride glides away from its emission point. `tickSound()` **re-triggers it every 115 ticks** (the sample's 5.77 s length ‚Äî Java's `sound_loop` cadence), each play preceded by a `stopsound` so exactly one instance can exist, and each re-trigger re-anchoring the emission at the rider (a 256-block distance guard also re-anchors mid-cycle at extreme speeds). For **200 ticks after any player join** (`playerSpawn` `initialSpawn` ‚Üí `lastJoinAt`) the cadence drops to 20 ticks: a play emitted while the client is still on its loading screen is dropped with no way to detect it, and believing that ghost copy for a full cycle was the "sound only starts seconds after loading in" bug ‚Äî on the warm cadence the first play the client actually receives lands within a second of the screen dropping, at the cost of a few stopsound-restart seams of a uniform rolling loop. Two failed designs bracket this one: a play-it-once version trusted the FSB's baked-in FMOD loop flag to run forever, but through a pack definition the loop did not reliably engage ‚Äî one 5.8 s play, then silence until the next 256-block re-anchor, and a play emitted at a just-(re)joining client was dropped outright, muting the ride while the script believed a loop was running (the fixed clock self-heals both within one cycle); and the *original* timer version re-triggered **without** the preceding stop ‚Äî whenever the loop flag *did* engage, every cycle stacked another immortal copy into a slowly-phasing hundred-cart chorus. The stop-then-play clock is immune from both directions: loop or no loop, at most one copy exists and a fresh one starts every sample length.
- **The RP shortens the hotbar item-name popup** (`ui/hud_screen.json`). Scrolling off a named hotbar item onto an empty slot leaves the old name fading out on vanilla's schedule (hold 1 s, then a 2 s `in_expo` fade ‚âà 3 s of lingering text) ‚Äî the popup has no "selected slot is now empty" signal, so it cannot be *cancelled*, only shortened. The pack overrides just the four stay/fade animation elements (JSON-UI files merge per-element with vanilla's; everything else in `hud_screen.json` stays untouched) to hold 0.5 s and fade 0.3 s, so a name is gone ~0.8 s after the last scroll. Global while the RP is active (every item-name popup, not just the pinned items); if a game update renames the elements the override silently no-ops ‚Äî safe failure.
- **Requires Bedrock 1.21.120+** (`@minecraft/server` module `2.3.0`, `min_engine_version [1,21,120]` ‚Äî `dimension.getBiome` and the `minecraft:redstone_producer` block component, both 1.21.120-era, are the gates). Both pins can be raised freely for newer-only targets.
- **Rails are decorative for physics.** No entity rides the physical rails on Bedrock (the pace is virtual, the ride cart is velocity-driven), but the track is still built from genuinely powered golden rails on redstone blocks, so it works for manual minecart rides after `stop`.
- **`/reload` reloads both functions and scripts** on Bedrock; the script re-initializes lazily and resumes the ride from its persisted state. Editing `config.mcfunction` + `/reload` refreshes knobs mid-ride, same as Java.
- **Only players generate terrain on Bedrock.** There is no working equivalent of Java's `forceload`-driven generation: `/tickingarea` keeps already-active chunks ticking but generates and pre-loads *nothing* (two corridor designs built on it failed identically ‚Äî the builder crawled along the rider's own simulation bubble, building in bursts right in front of the cart). The pack's answer is the **chunk scout** (`infinite_rail:scout`): an invisible entity whose vanilla `minecraft:tick_world` component makes it a mobile 6-chunk ticking area. It glides ahead of the rig ‚Äî stepping only onto ground whose chunk is already open, so it can never strand itself ‚Äî and between the rider's bubble and the scout's, the corridor from the rig to ~`.PACE_CART_BEHIND` blocks past the pace stays loaded and script-readable. How far terrain actually *generates* ahead is governed by the rider's **render distance** (the scout can only hold open what the engine has generated), which therefore needs to comfortably cover the corridor ‚Äî ~20‚Äì24 chunks at the default `.PACE_CART_BEHIND`; anything much higher just makes the generator churn forever behind a ride that never builds past `.PACE_CART_BEHIND` anyway.
- **"Loaded" and "ticking" are different states at the bubble's edge.** In the border ring around a ticking area, block lookups can succeed and hand back a `Block` whose *property reads* then throw `LocationInUnloadedChunkError`. The surface probe is therefore wrapped whole: any throw anywhere inside it reads as "no data at this column yet" and the sample falls back to the rolling average, instead of aborting the column (which used to stall the head at full gap and spam build errors).
- **The scout is a real simulation load**: its 13√ó13-chunk bubble ticks like an extra player at simulation distance ~6 (mob spawning included). This is the price of far-ahead building. The world's own simulation distance can (and should) stay at 4 ‚Äî it contributes nothing to the ride anymore, and every notch above 4 ticks hundreds of additional chunks around the rider for nothing.
- **Why Bedrock rides see fewer close mobs than Java (background for future ambience work ‚Äî no code change yet).** Bedrock's natural hostile-mob spawn/despawn sphere scales with the **player's simulation distance**: at sim 4 (what the pack's Tips recommend) mobs spawn only within ~44 blocks of the player and **despawn immediately past ~44 blocks**, while sim 6+ opens the sphere toward Java's ~128-block behavior. At ride speed (8‚Äì32 blocks/s) a mob spawned in that thin 24‚Äì44-block shell is carried out of the sphere and hard-despawned within a couple of seconds ‚Äî before it can path to the ~16-block aggro range ‚Äî which is why Java rides get chases and Bedrock rides get distant cameos. Compounding it: the track's light block suppresses spawns near the line (see the Track light mode ‚Äî `mode_light_off` darkens new tunnels and lets mobs spawn in them), and the scout's far-ahead bubble spawns its share of the mob population ~120 blocks ahead of the rider. Known levers, deliberately not pulled yet: recommending sim distance 6+ in the Tips (costs simulation perf), or a scripted ambience spawner (rejected for now as too much complexity).
- **The builder tolerates a lagging frontier.** A column needs only its own chunk plus a one-chunk margin (`BUILD_MARGIN`, 17 blocks ‚Äî at least 4 of the 12 window samples) loaded to build; missing far samples fall back to the rolling average *individually* (`badSamples` in the debug line). The guard exists to prevent deciding columns with **zero** real samples (which would freeze the average and bake a flat line into the world) ‚Äî requiring the entire 48-block window, as the port originally did, pinned the head ~49 blocks behind the frontier and caused the bursty, build-only-when-close behavior. While the builder is starved anyway, the pace **eases off smoothly** (the allowed speed shrinks with the remaining track buffer) rather than letting the ride outrun the track. If starvation persists, a one-time chat warning points at debug mode (`/function infinite_rail/debug`), which reports the loaded frontier, the scout's lead over the head, and the algorithm's live numbers (`badSamples`, `avg`, `railY`) every 16 blocks.
- **The scoreboard bridge self-heals.** The startup self-test verifies that API-written scores are visible to commands; if a version splits the two scoreboards, the script switches to a command-based bridge (inputs via `/scoreboard`, the brain's answer read back through execute-if-score successCount probes) and says so. In that mode live `.KNOB` tweaks read as config defaults.
- **Distribution is a single `.mcaddon`** (behavior pack + the small resource pack holding the invisible client definitions of the seat and scout entities); the BP's manifest depends on the RP, so activating the BP pulls the RP in automatically.
- **A startup self-test** exercises the script‚Üîcommand scoreboard bridge and the shared `decide` function once per load (when no ride is active) and reports loudly and specifically if either leg is broken, instead of letting the ride degrade into a silent flat line.
- **Rig integrity is self-healing**: duplicate seats/carts/scouts from rejoin races are removed on sight, a missing seat gets a 2-second grace period (so a merely-still-loading original isn't duplicated) before the rig is rebuilt, while a missing cart prop heals cart-only after ~1 s without unseating the rider, a missing scout is respawned at the rig (the one spot the rider guarantees is loaded) and walks itself back to its post, and the ride freezes entirely while its rider is offline.
- **The rider is in survival mode, not adventure.** Bedrock does not naturally spawn mobs around adventure-mode players (Java's spawning ignores game mode), which left the whole ride lifeless ‚Äî no animals, no monsters. Survival restores normal spawning; the rider stays untouchable through Resistance 255 + the damage gamerules, `mobGriefing false` protects the track, the per-tick inventory clear leaves nothing to place or swing, and neither rig piece can be entered. Old saves whose rider is still in adventure resume fine (the keeper accepts both modes).
- **Invisibility = "mobs can't see you" = "the arm is hidden" ‚Äî one lever, now owned by the Mobs-aggro mode.** The old `.HIDEHAND` knob kept a permanent invisibility effect on the rider to hide the first-person arm (`/hud` cannot touch the hand). On Bedrock, though, invisible players are **completely undetectable by hostile mobs** ‚Äî no sneaking creepers, no bow-draws, no chases ‚Äî which is why every Bedrock ride used to glide through the night in dead silence while Java rides got the full ambience. The two consequences are physically inseparable, so the knob is retired and the invisibility now follows `.AGGROMODE` (¬ß6.9): aggro **on** (the default) = visible = mobs react and the arm shows; aggro **off** = invisible = mobs ignore the ride and the arm (plus the rider's F5 body) is hidden ‚Äî with the inventory always empty, nothing renders. The keeper asserts the current state once a second; `stop` clears the effect with the others.
- **Single scripted rider:** the ride belongs to the player who started it (or the first player, on auto-start); only that player is re-seated by the keeper. Leave the ride the sanctioned way ‚Äî switch to creative or run `stop` ‚Äî exactly like Java.
- **Ride modes are score-driven on the Bedrock side too.** The `mode_*` functions only run commands and flip `.RAINMODE`/`.NIGHTMODE`/`.TORCHMODE`/ `.SKYMODE`/`.AGGROMODE`; the script never needs a scriptevent for them. It reads `.SKYMODE`, `.AGGROMODE`, the tri-state `.TORCHMODE` and the Track light `.LIGHTMODE` (`lightLevel()`, cached ~1 s on cmd-bridge worlds where non-preset levels degrade to the nearest preset) through the same bridge as the brain flags (native API normally, successCount probes cached ~1 s on cmd-bridge worlds), asserts the adjustable sky cruise `.skyspd` (default `.SKYSPEED`) each tick while sky mode is on, resets the ocean counters on the sky-off transition, asserts the rider's invisibility to match the mobs-aggro mode once a second (the keeper), and plants torch-mode torches from `maybeTorch()` in the column builder ‚Äî gated by `torchLit()`, which in the default auto mode hands `world.getTimeOfDay()` to the shared `torch_auto` and reads back `.torchlit` (cached ~1 s). Rain and the tri-state time functions use Bedrock's stable lowercase gamerule names directly in the function files.
- **The pinned hotbar items open native forms / nudge the speed.** The inventory keeper pins seven named items instead of clearing their slots ‚Äî the "Ride Settings" (chest minecart), "Visual Settings" (soul campfire), "Tips" (book) and "Debug" (smithing table) menu items plus the "Speed ‚àí", "Speed Reset" and "Speed +" items, mirroring Java's hotbar layout (Reset dead center at slot 4) (deliberately *not* slot-locked ‚Äî `ItemLockMode` items get a lock badge and a "Can't be moved/dropped/‚Ä¶" tooltip block, and the per-tick keeper makes the lock redundant anyway). **The speed trio are the pack's own custom items** (`bp/items/speed_*.json` ‚Üí `infinite_rail:speed_down/reset/up`), pure non-placeable icons whose sprites the RP's `textures/item_texture.json` maps onto the vanilla rail / minecart / powered-rail item textures (the same no-texture-shipped trick as the block disguises; `en_US.lang` names them for completeness, though the keeper always nameTags them). They used to be the real vanilla items ‚Äî which are **placeable**, so every click aimed at a block within reach (the track sits right under the crosshair) had the client *predict* the placement (the minecart item onto the powered rail below was the worst: the hotbar stack visibly emptied) before the cancelled interact event rolled it back with a hotbar resync, and a resync racing the mouse wheel **eats scroll input** ‚Äî the "scrolling around the middle Speed item needs two notches" bug. A custom item with no block gives the client nothing to predict: no rollback, no resync, no eaten scrolls. The script probes once per session whether the custom ids resolve (`pinnedItemType`) and falls back to the old vanilla items on a mismatched/outdated pack pair ‚Äî the use handlers match both, so a stale save's old items keep working until the keeper swaps them. Because the menu items are still placeable blocks/carts chosen for their icons, one click can arrive through two events ‚Äî `world.afterEvents.itemUse` (aiming at air) and `world.beforeEvents.playerInteractWithBlock` (aiming at a block, **cancelled** so the survival rider can never build the icon into the world ‚Äî and so a pinned-item click can't open a passing chest/door, with the real action deferred out of the before-event's read-only window; this is the `@minecraft/server` 2.x replacement for the removed `itemUseOn` events ‚Äî subscribing to a nonexistent signal throws at module load and kills the whole script, so the subscribe is additionally guarded) ‚Äî so both paths funnel through one handler behind a debounce, matched by item type + name + rider. **The debounce is two-tier**: 2 ticks for the speed items ‚Äî the smallest window that still folds a click's duplicate event pair (itemUse + the deferred interact, at most a tick apart) into one action, so distinct clicks and held-button refires land as fast as the client sends them, like Java's stat-counted items ‚Äî and 4 ticks for the menu items, so a double-delivered click can't pop a form open twice. The Ride Settings item shows a `@minecraft/server-ui` **ModalFormData** (the sky, cart-sound, hide-minecart and mobs-aggro toggles ‚Äî the aggro label names the arm consequence, since on Bedrock the same invisibility that blinds mobs is what hides the first-person arm ‚Äî then a 1‚Äì64 speed slider at the bottom whose label names the active config default; the Speed + item goes beyond the slider's 64) pre-set from the live scores, with only actual changes running their function files (so the tellraw feedback matches the chat commands exactly ‚Äî the slider feeds the shared `speed_step` as a delta computed against the live *active* speed via `activeSpeed()`, not the clamped 64-max slider display ‚Äî an over-64 ride keeps its speed unless the slider is actually moved). **The slider tunes whichever cruise is active** ‚Äî the sky cruise `.skyspd` in sky mode, the ocean cruise `.ocnspd` mid-sprint, else the land speed ‚Äî and its label names the matching default (`.SKYSPEED` / `.OCEANSPEED` / `.DEFAULTSPEED`); the Visual Settings item shows its own ModalForm (rain, the tri-state Time dropdown, the tri-state Torches dropdown ‚Äî Off / On day-and-night / Auto at night, index = the `.TORCHMODE` value ‚Äî the Low/Medium/High/Max torch-density dropdown ‚Äî friendly names only ‚Äî and the Off/Low/On **Track light** dropdown, running the same `mode_torches_*` / `torch_density_*` / `mode_light_*` files as Java's books); the Tips item shows a read-only **ActionFormData** with the recommended settings (hide the HUD, FOV 100+, lowest simulation distance, 16-24+ render distance) plus the Bedrock-specific ones (disable "Enable Game Pause", "Show Pause Menu on Focus Lost" and "Lower Framerate when Controller is Disconnected" under Settings > General); the Debug item shows the debug form (chat-output toggle, the sidebar-view dropdown, a command-help printer); the Speed items run `speed_inc`/`speed_reset`/`speed_dec`. The BP manifest therefore depends on `@minecraft/server-ui` `2.0.0` (stable long before the pack's 1.21.120 floor). `stop` removes all seven.
